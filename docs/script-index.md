## Part 1 тАФ Original vision (handwritten)

This is to be the canonical list of capsule-pro pnpm scripts that i want to exist, defined by their actual real functionality. AI agents MUST ensure that the scripts assigned to this index exist in FULL, with a complete explanation of how exactly they are accomplishing what I am defining here, not just the commands, a full breakdown of ALL the commands and configurations being used to construct the script.

**1. `pnpm manifest:validate`** тАФ validate all the manifest soures in `manifest/source/`. This script should check all the sources to find out if they are valid format, ensure they have all the necessary definitions of entities with their full list of advanced functionality and configurations and relationships. It should also ensure that there are no unnecessary or annoying guards in place that do nothing except annoy the user or expose them to internal coding leaks. Users should never be forced to enter or find numbers or fields or anything that manifest itself can provide. This step should also ensure that manifest isn't ever asking for or requiring a field which should be auto generated by manifest, or required when the user cant even have access to it.

**2. `pnpm manifest:compile`** тАФ compiles all the `.manifest` files into a single ir AND ensures that the single IR is being consumed everywhere it needs to be. It should normalize all case to the same type (the same type that the db will use) so that no code will ever drift or conflict due to naming, case, types, etc. it should also use similar rules like `packages/typescript-config/base` does, ensuring there aren't things like unused entities or relationships, or things that arent actually wired up properly, or fake implementation, or ai slop.

**3. `pnpm manifest:build`** тАФ brings everything together, connecting the code, the db, the ir, the API, hooks, ui elements. It should bridge everything to be interconnected.

The upstream manifest CLI is single-file тЖТ single-surface тЖТ one artifact per invocation. Capsule-pro is the opposite: 104 `.manifest` files (103 domain + `_base.manifest`) across 13 domain folders, custom projection options, post-generation patching, a typed client the CLI doesn't emit, and CI gates that assert "committed code == freshly generated code." None of that is expressible through `manifest compile` / `manifest generate` alone, so the scripts are the orchestration + glue layer.

### Vision тЖТ command mapping

- **Part 1 ┬з1 `manifest:validate`** тЖТ folded into `manifest:ci` (owner directive 2026-07-11: "validate should just be part of ci"). The husky pre-commit hook still validates staged `.manifest` changes directly via `pnpm exec manifest validate manifest/ir/kitchen.ir.json`. The Part 1 gap (per-source validation, annoyance-guard detection) is unchanged тАФ tracked as a `full-ci.mjs` improvement, not a separate script.
- **Part 1 ┬з2 `manifest:compile`** тЖТ step 1 of `manifest:build` (`compile.mjs` тАФ native `compileProjectToIR` merge, topo-sort, deterministic `irHash`, idempotent provenance; writes `kitchen.ir.json` + commands registry + provenance + merge report + module graph).
- **Part 1 ┬з3 `manifest:build`** тЖТ `manifest:build`, now genuinely "brings everything together": it is the ONLY regen command and covers surfaces the old `manifest:sync-artifacts` missed (Prisma schema projection was previously a separate hand-typed CLI invocation).

### Known exclusions (documented, deliberate)

- **`manifest fmt`** тАФ NOT in either command. The upstream CLI parses each source file standalone and fails on cross-file references (`mixin TenantScoped` from `_base.manifest`) in 101/104 files. Blocked upstream until fmt resolves `use` imports. Do not wire it in; do not "fix" by removing mixins.
- **Advisory audits with open debt** тАФ `audit-dead-commands`, `audit-ir-drift`, `lint-schema`, `validate-ai`: not blocking gates; run directly (Part 3) when mining debt.
- **`check:manifest:structure` / `check:manifest:domain`** тАФ repo-level audits under `tools/`, not part of the manifest pipeline surface.

### Source of truth

```
manifest/source/**/*.manifest  тЖТ  compile  тЖТ  manifest/ir/kitchen.ir.json  тЖТ  projections
                                                                              тЖУ
                                                   schema / routes / stores / hooks / client
```

The IR is the single semantic source of truth. Prisma models, API routes, runtime stores, and UI hooks are projections of the IR тАФ not the other way around. All projections are **committed**; `manifest:ci` (and the CI workflow's regenerate + `git diff` job) prove committed == freshly generated.

### What Capsule scripts do that the upstream CLI cannot

- **Multi-module merge** тАФ `compile.mjs` + `compileProjectToIR` (CLI `--glob` is broken for 104 modules)
- **Domain route remapping** тАФ `generate.mjs` + `ENTITY_DOMAIN_MAP`
- **Post-generation patching** тАФ accessor config, field overrides, orphan cleanup
- **Typed Capsule client** тАФ `generate-capsule-client.mjs`
- **CI drift gates** тАФ byte-compare committed artifacts vs fresh regeneration

### Quick reference

**Edited `.manifest` source:**

```bash
pnpm manifest:build          # regenerate everything
# manifest.prisma changed? тЖТ docs/database/README.md migration workflow
pnpm manifest:ci             # all gates
# restart the API dev server (it boots on the embedded IR)
# commit source + regenerated artifacts together
```

**Compiler bump:**

```bash
pnpm manifest:update && pnpm install
pnpm manifest:build
pnpm manifest:ci
```

**Nothing changed?** Run nothing. Generation is a change-time concern; CI re-proves the world on every PR.
