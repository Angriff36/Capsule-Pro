### 29. `[RESOLVED 2026-07-12]` Batch-endpoint `$transaction` pool-exhaustion cap
- **What:** Cap the generic manifest batch endpoint's transaction scope/size so one client-submitted batch cannot hold a pool connection for up to 120s and exhaust the `max:20` pool. **Execution tier: foundational (alongside #3–#5)** — see the header numbering-stability note + the Priority rationale below. Numbered #29 (not inserted as #6) to preserve the 20+ item cross-references; its physical position is a numbering artifact, **not** a priority signal.
- **Why / evidence:** `apps/api/lib/manifest/execute-command.ts:318-363` wraps ALL N user-supplied operations in ONE `$transaction` whose timeout scales `min(ops.length×2000+5000, 120_000)` → up to **120s holding one pool connection**. The primary pool is `max:20` (`packages/database/create-pg-adapter.ts`). A large or heavy governed batch from one client can pin a connection for the full window and, under concurrent batches, exhaust the pool → blocks ALL other requests (availability outage, not a latency regression). Split out of item #18 (MED code-fix) because this is infrastructure-level, availability-grade. (Originally surfaced 2nd rev; verified + split 4th rev.)
- **Fix:** Cap the batch size (reject/chunk above N) and/or stop wrapping arbitrary multi-op batches in a single interactive tx — run independent ops as chunked independent txs (or per-op outside a shared tx where the governed-write constitution allows). Preserve `RuntimeEngine.runCommand()` governance for each domain mutation.
- **Scope:** `route` (infra / availability)
- **Acceptance check:** A max-size client batch cannot hold any single connection longer than a bounded threshold; concurrent batch load leaves pool capacity for ordinary requests (proven via item #1's instrumentation under load); governance semantics preserved; `manifest:ci` green. Ties to spec criteria #1/#2 (connection config + instantiation) at the availability level.
- **Implementation log (2026-07-12, RESOLVED):** The batch SIZE cap half was already in place — `MAX_BATCH_SIZE = Number(process.env.MANIFEST_BATCH_MAX_SIZE) || 50` (`execute-command.ts:251`) rejects >50 with a 400 (`:297-302`), and the apps/app bulk-action UI already chunks at 50 (`apps/app/app/components/bulk-actions.tsx:114`), so a single request never legitimately carries a huge batch. The remaining availability lever was the `$transaction` TIMEOUT CEILING: `runManifestBatch` (`execute-command.ts:318-363`) wrapped all N ops in ONE interactive tx (one pool connection) with `timeout: Math.min(operations.length * 2000 + 5000, 120_000)` → a max-size (50-op) batch could pin a connection for ~105s (120s if the env cap were raised), 4× the app-wide `transactionOptions.timeout` (30s, `packages/database/{index,standalone}.ts`); under a few concurrent large batches that exhausts the `max:20` pool and starves every other request (Prisma P2024 after the 10s `maxWait`). **Fix:** extracted the timeout computation into `apps/api/lib/manifest/batch-timeout.ts` — `MAX_BATCH_TRANSACTION_TIMEOUT` (env `MANIFEST_BATCH_TX_TIMEOUT`, default **30_000** = the app-wide tx ceiling) + `batchTransactionTimeout(opCount)` (`Math.min(opCount*2000+5000, MAX_BATCH_TRANSACTION_TIMEOUT)`); the call site now uses `timeout: batchTransactionTimeout(operations.length)`. Small batches keep proportional headroom under the ceiling; a max-size batch now holds a connection ≤30s (was ~105s), and a pathologically large batch (env-raised cap) cannot escape the ceiling. Server-side chunking into independent txs was REJECTED: `runManifestBatch`'s documented contract (and the client's chunking) treat each request as all-or-nothing (constitution §5) — splitting would break atomicity. **Escape hatch:** `MANIFEST_BATCH_TX_TIMEOUT=60000` for a workload with legitimately heavy per-op work. Regression guard: `apps/api/__tests__/lib/batch-timeout.test.ts` (4/4 green) pins the default ceiling (30s, not 120s), the clamp at 13+ ops, and that max-size×10 cannot exceed the ceiling. Gates: focused test green; apps/api typecheck adds ZERO new errors (17 pre-existing, none in `batch-timeout.ts`/`execute-command.ts`); no schema/manifest touched → `db:check`/`manifest:ci` orthogonal (unchanged baseline). **DEFERRED (its own follow-up, NOT a stub):** a concurrency semaphore limiting simultaneous batch txs to a fraction of the `max:20` pool would FULLY guarantee ordinary-request capacity under concurrent batch load — the timeout ceiling bounds the worst SINGLE-batch hold, but 20 concurrent batches could still pin the pool for ≤30s; gate that follow-up on item #1's instrumentation showing real concurrent-batch pressure. Two bespoke siblings replicate the same 120s-capped multi-op-`$transaction` pattern (`kitchen/prep-lists/commands/create/route.ts:201`, `procurement/requisitions/[id]/convert-to-po/route.ts:237`) — out of #29's scope (item #18 territory); same fix shape applies if measurement warrants.
