## Summary

Two coupled changes:

1. **Bump `@angriff36/manifest` 0.5.0 → 0.6.1** across all six `package.json` files (root, apps/api, apps/app, packages/manifest-adapters, packages/manifest-ir, packages/mcp-server) and refresh `pnpm-lock.yaml`. This pulls in the new `manifest enforce-surface` CLI command plus the v0.6.1 format-tolerant registry loaders.

2. **Add a `manifest-enforce-surface` job to `.github/workflows/manifest-ci.yml`** as a **reporting gate** (`continue-on-error: true`, no `--strict`). Contributors see findings in PR check status without being blocked.

## Why reporting, not blocking?

Per the agreed roadmap, the gate stays advisory until:

- Upstream Manifest ships raw SQL / `$executeRaw` / `$queryRaw` write detection (next dedicated PR upstream).
- The 30 known true-positive findings in this tree are either fixed or have explicit bypass-registry entries.

Once both land, this PR's only required change is flipping `continue-on-error: false` and adding `--strict` to the run step.

### What each bucket means here

| Code | Action |
|---|---|
| `ROUTE_SURFACE_DRIFT` (24) | Concrete per-command routes outside `/api/manifest/[entity]/commands/[command]`. Each either migrates to the canonical dispatcher or gets a `DEPRECATED ALIAS` banner in the file. |
| `EXISTING_COMMAND_AVAILABLE` (3) | `createInventoryTransaction` in `apps/api/app/api/shipments/[id]/status/route.ts` (uses `database.$executeRaw INSERT`), `createEvent` in `apps/api/app/api/events/import/server-to-server/route.ts` (same), `_createBattleBoard` in `apps/api/app/api/events/documents/parse/route.ts` (uses `database.battleBoard.create({})` — Prisma aliased to `database`). All bypass registered `Entity.create` commands. |
| `DYNAMIC_COMMAND_UNVERIFIABLE` (3) | The canonical dispatcher `apps/api/app/api/manifest/[entity]/commands/[command]/route.ts` (expected — the dispatcher's whole point is dynamic) plus two routes with dynamic command names that should ideally use a typed wrapper. |
| `BYPASS_VIOLATIONS_NO_REGISTRY` (1) | Informational: no `--bypass-registry` path was supplied. When we author `bypasses.json` for the exempt direct writes, this warning disappears. |
