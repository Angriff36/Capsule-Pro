# Capsule-Pro-Convex Constitution

## Manifest Projection Charter

Version: 1
Status: Binding
Supersedes: Capsule-Pro Constitution v4 (runtime + store-adapter model) for this repo
Audience: Humans, agents, CI checks, repo auditors, implementation plans

---

## 0. Purpose

This constitution defines how **capsule-pro-convex** integrates with Manifest.

Its purpose is to remove architectural ambiguity and lock one architecture in stone.

**The core inversion vs. Capsule-Pro v4:** governed behavior is **compiled, not interpreted.**
There is **no `RuntimeEngine.runCommand` in the request path** and **no Manifest store adapter.**
The Manifest **Convex projection** compiles the domain (policies → guards → constraints →
mutate → semantic event → reactions) directly into the generated Convex backend. The
generated `convex/` code **is** the governed runtime. Convex executes it; Convex persists it;
Convex makes it reactive.

The single source of domain truth is the compiled Manifest **IR**. Everything in `convex/` is a
**deterministic projection of that IR** and must be regenerable from it. The lock that makes
this binding is the **drift gate** (§6): committed `convex/` output must byte-for-byte equal a
fresh regeneration from IR, or CI fails.

Agents and contributors must not ask whether governed domain mutations should go through the
Manifest-generated Convex mutations. They must. If existing code disagrees, the code is
nonconforming unless it is explicitly listed in the approved bypass registry.

Capsule-pro-convex integrates with Manifest. Capsule-pro-convex does not redefine Manifest
semantics, and it does not hand-author governed mutations.

---

## 1. Normative Authority

Normative authority for governed domain semantics resides in:

1. Capsule-owned Manifest source — `manifest/source/**/*.manifest`
2. The compiled IR generated from that source — `manifest/ir/*.ir.json`
3. The Manifest package contract for the Convex projection — `@angriff36/manifest`
   (`docs/spec/ir/ir-v1.schema.json`, `docs/spec/semantics.md`, `docs/spec/adapters.md`,
   and the `convex` projection surfaces)
4. The generated Convex backend regenerated from (2) — `convex/**`

Order of authority when sources disagree: **IR source → compiled IR → generated `convex/`.**

If hand-edited `convex/` code conflicts with a fresh regeneration from IR, the hand edit is
wrong. If generated `convex/` conflicts with IR, IR wins and the projection/options must be
fixed and regenerated. The correct response to any conflict is to **change IR and regenerate**,
never to hand-patch generated output or route around it.

---

## 2. Definitions

**Governed entity** — any domain entity whose state carries business meaning: tenant,
operational, workflow, financial, scheduling, inventory, staff, customer, or supplier state.
In Manifest terms: an entity with a persistent `store` target (`durable` / `postgres` /
`supabase`). Tenant-scoped entities are governed by default. An entity is not "ungoverned"
merely because it currently lacks Manifest coverage — it is `unknown_nonconforming` (§8).

**Governed mutation** — any create / update / delete / transition / command that changes
governed domain state. A governed mutation exists **only** as a Convex `mutation` emitted by the
Manifest Convex projection (`convex.mutations` surface), carrying the generated header marker.

**Generated header marker** — the exact line every projected Convex file carries:

```
// GENERATED by the Manifest → Convex projection. DO NOT EDIT.
```

Its presence marks a file as authoritative-but-derived. Its absence on a file that performs
`ctx.db` writes is a bypass.

**Bypass** — any governed write that does not flow through a generated Convex mutation.
Bypasses are forbidden unless listed in `manifest/governance/bypasses.json`.

**Read path** — code that loads or projects state without changing governed truth. Read paths
may query Convex directly but must not define behavior, enforce invariants, invent state
machines, or synthesize semantic events.

**Semantic event** — a domain event representing the result of a successful governed mutation.
Semantic events may only be written by generated Convex mutations (the projected event-row
append). Operational logs, analytics, telemetry, and debug traces are **not** semantic events.

---

## 3. The Compile Invariant

Governed domain behavior exists in exactly one form in this repo: **Convex code generated from
IR by the Manifest Convex projection.**

- Governed behavior is **compiled** from IR, not **interpreted** at runtime. There is no
  `RuntimeEngine` instance, no `runCommand`, and no IR loaded into a live engine in any request
  path.
- The generated `convex/mutations.ts` already enforces, per command and fail-closed:
  policies → guards → constraints (runtime order) → mutate actions → semantic event row →
  matched reactions. This **is** the governance. It is not a convenience wrapper around it.
- Hand-authored governed behavior is nonconforming. If a governance rule needs to change, it
  changes in `.manifest` source → IR → regenerated `convex/`.

---

## 4. System Boundaries

### Clerk
Clerk is the identity and organization-context provider: who the actor is, which org/tenant is
active, the session, and Clerk-side roles/memberships. Clerk does **not** define domain
semantics, does not replace Manifest, and does not make a Convex mutation safe by itself.
Capsule-pro-convex must translate authenticated Clerk context into the **server-injected
arguments** a generated mutation requires (§9) before that mutation runs.

### Manifest
Manifest is the semantic authority. The `.manifest` source and compiled IR answer: what command
exists, what entity it governs, what it means, valid inputs, required tenant/actor context, the
policy/permission rule, legal transitions, invariants, effect boundaries, emitted semantic
events, and conformance evidence. Manifest's authority reaches this repo **through the projected
Convex code**, not through a hosted runtime.

### Convex
Convex is **both** persistence **and** the execution surface for generated governed mutations,
**and** the reactive query layer. Convex `ctx.db` writes are permitted **only inside generated
mutation files** (header-marked). Convex queries are permitted freely for read models,
projections, dashboards, and reporting, provided they do not define domain semantics (§10).
Convex schedulers (`crons`) and HTTP actions are emission targets of the projection
(`convex.crons`, `convex.http`), not hand-authored governance.

### Next.js routes / server actions
Transport surfaces only. They authenticate the actor, resolve tenant context, and **invoke a
generated Convex mutation** (directly via the Convex client, or via the external HTTP door in
§6). They must not own governed behavior, perform direct governed writes, or duplicate
transition/invariant/event logic.

### UI and Command Board
The UI may visualize state, collect intent, and orchestrate mutation calls. It must not encode
authoritative state machines, transition graphs, validation rules, or invariants. The Command
Board is an orchestration surface, not a semantic authority. Every governed mutation it triggers
must call a generated Convex mutation.

---

## 5. Canonical Write Path

The canonical governed write path is:

1. Request enters a route, server action, job, workflow, or the Command Board.
2. Capsule-pro-convex authenticates actor context (Clerk).
3. Capsule-pro-convex resolves tenant/organization context server-side.
4. Capsule-pro-convex calls the **generated Convex mutation** for the command, passing
   server-injected tenant/actor args (never client-trusted — §9).
5. The generated mutation runs compiled governance fail-closed:
   policies → guards → constraints → mutate actions → semantic event row → matched reactions.
6. Convex persists state and the event row in one transaction.
7. Matched reactions fan out **inline within the same transaction** (no outbox).
8. The mutation returns its result; diagnostics surface as thrown Convex errors.
9. The application surfaces the result and reacts to live Convex query updates.

No governed mutation may skip this path. There is no `RuntimeEngine.runCommand` step — steps
5–7 are the **compiled** equivalent, executed by Convex.

---

## 6. Generation and Drift — The Lock

All governed Convex code is generated from IR by `manifest generate --projection convex`. Six
surfaces are authoritative-derived and must be regenerable byte-for-byte:

| Surface | Path |
|---|---|
| `convex.schema` | `convex/schema.ts` |
| `convex.queries` | `convex/queries.ts` |
| `convex.mutations` | `convex/mutations.ts` |
| `convex.crons` | `convex/crons.ts` |
| `convex.http` | `convex/http.ts` |
| `convex.sagas` | `convex/sagas.ts` |

**The lock:** CI regenerates all six surfaces from committed IR into a scratch directory and
diffs against the committed `convex/`. Any difference is **drift** and **fails the build**
(`manifest:convex:check`). This is the mechanical enforcement of §1 (IR is authority) and §3
(behavior is compiled). "IR and the backend cannot disagree" is not a slogan here; it is a gate.

Generation is deterministic given the same IR, projection options, and `@angriff36/manifest`
version. The pinned package version is part of the drift contract: bumping it is a change that
must be regenerated and committed together (§14).

### External HTTP door
The `convex.http` surface MAY expose external/webhook callers. It is the **only** non-Convex-client
entry to a governed mutation, and it must do nothing but authenticate, map input, and
`ctx.runMutation` the generated mutation. No per-command bespoke REST routes for governed
commands may be hand-authored. *(Flagged default — disable this surface if no external callers
exist; do not keep a door you don't use.)*

---

## 7. Direct Write Prohibition

Capsule-pro-convex must not mutate governed entities outside a generated mutation file.

A **direct write** is any of the following in a file **without** the generated header marker:

- `ctx.db.insert` / `ctx.db.patch` / `ctx.db.replace` / `ctx.db.delete`
- a direct `ctx.runMutation` to a non-generated mutation that writes governed state
- any helper, repository, or wrapper that ultimately performs the above

Forbidden source surfaces for governed writes: Next.js routes, server actions, React
components, background jobs, crons authored by hand, workflow orchestrators, import/export
scripts, test helpers, seed scripts, admin tools, and one-off repair scripts.

Renaming a write behind a helper does not make it conforming. Exception: writes listed in
`manifest/governance/bypasses.json` (§8).

---

## 8. Governed Entity & Bypass Registries

`manifest/governance/entities.json` classifies every entity as exactly one of:
`governed` · `read_only_projection` · `infrastructure` · `bypass_allowed` · `unknown_nonconforming`.

Tenant-scoped entities default to `governed`. `unknown_nonconforming` is technical debt, not
permission to continue a pattern.

`manifest/governance/bypasses.json` lists every approved direct write, each with: entity, file
path, reason the bypass is safe, why governance is not required, the tenant/security boundary
that still applies, owner, approval date, and review/expiry date. Agents must not invent
bypasses; an unlisted direct write is a violation, not a judgment call.

---

## 9. Tenant Context Rule (Projection-Specific)

Convex has **no row-level security.** Tenant isolation is therefore a hard requirement of this
constitution, enforced two ways:

1. **Server-injected mutation args.** The tenant/org id passed to a generated mutation must be
   derived from authenticated Convex/Clerk context on the server. It must **never** be trusted
   from the client. A mutation argument that accepts a client-supplied tenant id for a governed
   write is nonconforming.
2. **Tenant-scoped reads.** Generated queries scope by the tenant column via `withIndex`. Read
   models and dashboards must filter by the authenticated tenant; an unscoped governed read is
   nonconforming.

If Clerk context and the requested tenant disagree, the mutation must fail closed.

---

## 10. Read Path Freedom

Read paths may query Convex directly (dashboards, reports, projections, list/detail, search,
analytics, denormalized read models, server components). Reads must respect tenant isolation
(§9) and must not: mutate governed state, define transitions, enforce alternate rules,
synthesize semantic events, become hidden command paths, or treat projection/cache state as
semantic truth. Generated artifacts, MCP descriptors, Agent SDK schemas, generated clients, and
reports are **derived surfaces** — never semantic authority. If any disagrees with IR or the
generated backend, IR wins and the producer/projection is fixed and regenerated, never hand-edited.

---

## 11. Reactions, Schedules, and Outbox

- **Reactions** are compiled as **inline transactional fan-out** inside the emitting generated
  mutation. They run in the same transaction as the source write. There is **no outbox table**
  and **no sync cron** — Convex reactivity replaces them.
- **Schedules** declared in IR (`schedule`) are projected to Convex `cronJobs()` referencing
  generated mutations. Hand-authored crons that perform governed writes are forbidden.
- Capsule-pro-convex must not synthesize semantic events outside a generated mutation. If a
  mutation fails or rolls back, its semantic event and reactions roll back with it.

---

## 12. Forbidden Architecture (No Adapter, No Engine)

The following are **forbidden artifacts** in this repo. Their presence is nonconformance:

- A Manifest **store adapter** / `storeProvider` / any implementation of the runtime `Store`
  interface. (Persistence is owned by Convex via generated mutations, not by a runtime adapter.)
- A hosted `RuntimeEngine` or any `runCommand` call in a request path.
- IR loaded into a live engine for governed execution.

This repo is **projection-only**. The interpreter path (runtime + adapter) described by
Capsule-Pro v4 is explicitly retired here.

---

## 13. Forbidden Persistence (No Prisma)

Convex is the sole persistence and the sole projection target for governed state. The following
are **forbidden** in capsule-pro-convex:

- Prisma, `@prisma/client`, `prisma` CLI usage, `@repo/database`, Prisma migrations.
- Prisma-derived projections and their generators: prisma-store, prisma-model-metadata,
  kysely, drizzle, materialized-views-from-prisma, and Prisma-shaped zod/react-query generators.
- Neon/Postgres migration scripts (`db:neon-*`, `db:dev`, `db:deploy`, `migrate*`) for governed
  state.

If a read model needs SQL-shaped denormalization, it must be expressed as a Convex query or a
Manifest projection off IR — not a resurrected Prisma path.

---

## 14. Change Protocol

Any change to governed semantics must follow this order:

1. Edit `.manifest` source.
2. Recompile IR (`manifest:ir`).
3. Regenerate all six Convex surfaces (`manifest:convex`).
4. Regenerate governance registries / drift baselines as needed.
5. Update read models / UI projections.
6. Update tests and run the governance gate (`manifest:ci`).

Capsule-pro-convex must not implement governed behavior first in `convex/` and retrofit IR
later. Temporary divergence, if unavoidable, must be recorded as documented nonconformance with
owner, reason, risk, and removal plan.

---

## 15. Testing and Conformance

- Tests for governed behavior must drive the **generated Convex mutations** (e.g. via
  `convex-test` / the Convex test harness), asserting state changes, emitted semantic events,
  reaction fan-out, and fail-closed denials.
- Tests must not write governed state directly to simulate a successful command.
- The drift gate (§6) is part of conformance: a green test suite over drifted output is not
  conformance.
- Test helpers that bypass generated mutations for governed behavior are nonconforming unless
  explicitly marked infrastructure setup and isolated from behavior assertions.

CI must include checks for: Convex drift (§6), direct governed writes (§7), hand-authored
mutations missing the generated header, bypass-registry violations, tenant-unscoped governed
reads/writes (§9), forbidden store-adapter / `RuntimeEngine` artifacts (§12), and forbidden
Prisma artifacts (§13).

## 16. Agent Operating Rules

Before changing code, classify the task: `governed mutation` · `read path` · `UI orchestration`
· `generation/projection` · `infrastructure/bypass` · `test/conformance` · `unknown`.

- **Governed mutation** → change `.manifest` source and regenerate. Never hand-edit `convex/`.
- **Read path** → direct Convex queries allowed; preserve tenant isolation (§9).
- **UI orchestration** → call generated mutations; do not implement rules.
- **Generation/projection** → change projection options/source; keep output deterministic.
- **Infrastructure/bypass** → verify `bypasses.json` before writing.
- **Unknown** → inspect `.manifest` source, IR, generated `convex/`, registries, and the
  Manifest spec. Do not infer semantics from route names. Do not ask the user to re-decide the
  architecture — it is decided here.

When code conflicts with this constitution, report:

```
NONCONFORMANCE:
- file:
- behavior:
- violated rule:
- safer replacement:
- whether patch was applied:
```

Agents must not: preserve nonconforming patterns because they already exist; hand-edit
generated `convex/` output; author concrete per-command governed routes; duplicate command
behavior in app code; invent Manifest semantics from route names; or treat Clerk permissions,
Convex helpers, or tool availability as a substitute for a generated governed mutation.

- `manifest/source/**/*.manifest` — Capsule-owned domain source.
- `manifest/ir/*.ir.json` — compiled IR (generated).
- `convex/{schema,queries,mutations,crons,http,sagas}.ts` — generated governed backend.
- `manifest/governance/entities.json` — governed entity registry (§8).
- `manifest/governance/bypasses.json` — approved bypass registry (§8).
- `manifest/governance/projection-options.json` — frozen Convex projection options (part of the

- `.github/workflows/manifest-governance.yml` — the CI gate.
- `manifest/scripts/*` — generation + gate scripts (§ replacement set).

Missing artifacts are **missing evidence**, not permission to guess. Where mechanical
enforcement does not yet exist, classify it as `MISSING ENFORCEMENT` and recommend the smallest
artifact that makes the rule machine-checkable.

## 18. Implementation Conflict Rule

Existing code is not automatically authoritative.

- The existence of a direct Convex mutation does not prove the write is allowed.
- The existence of a route does not prove it is canonical.
- The existence of generated code that differs from a fresh regeneration proves **drift**, not intent.
- A passing test that bypasses generated mutations does not prove conformance.
- The existence of Clerk auth does not prove domain authorization.

If implementation contradicts this constitution, the implementation is legacy or nonconforming,
and the fix is to bring it into conformance — change IR, regenerate, and pass the gate.

---

## 19. Plain Terms

Manifest defines what governed changes mean. The Convex projection compiles that into the
backend. Convex runs and persists it, reactively. Clerk says who is acting. Next.js receives
requests. The UI displays and orchestrates.

Only Manifest-generated Convex mutations govern domain writes. Reads can be flexible. Writes
cannot. No store adapter. No `RuntimeEngine`. No Prisma.

If a governed write does not go through a generated Convex mutation, it is either an approved
bypass or a bug. If the committed backend does not match a fresh regeneration from IR, it is
drift. If the repo disagrees with this constitution, fix the repo.
