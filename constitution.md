
# Capsule-Pro Convex Clone — Constitution

Manifest Integration Charter (compile-to-Convex)

Version: 5  
Status: Binding  
Scope: **capsule-pro-convex clone only** — not the upstream Manifest language repo  
Audience: Humans, agents, CI, auditors

---

## 0. Purpose

This repo implements Capsule Pro as **Manifest IR compiled to Convex**. Domain semantics live in `.manifest` source; persistence, governed writes, reactions, schedules, webhooks, and sagas are **emitted as Convex code**. Convex owns storage and reactivity.

The goal is **one backend truth** — not Manifest runtime + Prisma + Next.js API routes in parallel.

---

## 1. Two architectures (only one is allowed here)

Manifest supports two integration models. This clone uses **Path A only**.

### Path A — Compile-to-Convex (THIS REPO)

| Stage | Artifact |
|-------|----------|
| Author | `manifest/source/*.manifest` |
| Compile | `manifest/ir/kitchen.ir.json` |
| Project | `@angriff36/manifest/projections/convex` → `convex/` |

Governed writes are **generated Convex mutations**. Each mutation embeds the command pipeline:

```txt
policies → guards → constraints (fail-closed) → mutate → event row → inline reactions
```

**No `RuntimeEngine` in the request path.**  
**No `storeProvider` / Store adapter.**  
**No `@repo/database` / Prisma.**

The invariant is **structural**: the only write door is generated code marked `GENERATED by the Manifest → Convex projection`.

### Path B — Runtime interpreter (FORBIDDEN in this clone)

Live `RuntimeEngine.runCommand()` + `storeProvider` + Prisma/Postgres/custom adapter. Used when IR must be evaluated at runtime without regeneration (e.g. tenant-editable rules). **Not this repo.**

If you find Path B code here, it is **legacy** — delete or port, do not extend.

---

## 2. Governed mutation invariant

A **governed mutation** is any create/update/delete/command that changes tenant or domain state.

**Rule:** Governed mutations execute only through **generated Convex mutations** in `convex/mutations.ts` (and saga steps in `convex/sagas.ts` where emitted).

`runCommand` is the **semantic invariant**, not a function you must call at runtime. The Convex projection compiles `runCommand` semantics into each mutation handler.

### Forbidden bypasses

Direct writes include:

- `ctx.db.insert` / `patch` / `replace` / `delete` in hand-written Convex code
- `@prisma/client` or `@repo/database` mutations
- `createManifestRuntime`, `prismaOverride`, `storeProvider`
- Raw SQL against governed tables
- Next.js API routes that mutate domain state outside generated Convex functions

Renaming a bypass behind a helper does not make it conforming.

### Reads

**Convex queries** (`convex/queries.ts`) are read surfaces. They may bypass Manifest runtime but must:

- Preserve tenant isolation
- Not define domain semantics or state transitions
- Not emit semantic events

---

## 3. Reactions, schedules, webhooks

| Mechanism | How it works in this repo |
|-----------|---------------------------|
| **Reactions** | **Inline fan-out** inside the emitting mutation's transaction — same handler runs matched reaction target commands via `ctx.db.insert/patch`. Atomic with the source write. **Not** separate schedulers or triggers. |
| **Schedules** | IR `schedule` → `convex/crons.ts` via `cronJobs()` |
| **Webhooks** | IR webhooks → `convex/http.ts` |
| **Sagas** | IR sagas → `convex/sagas.ts` |

"No outbox" is true for reactions: fan-out is synchronous in the mutation, not eventual via outbox polling.

---

## 4. System boundaries

### Clerk

Identity and session only. Clerk does not define domain semantics. Map Clerk context into Convex auth / mutation args before governed writes. Clerk alone does not make writes safe.

### Manifest

Source of truth for **what** commands exist, **what** they mean, policies, guards, events, reactions. Expressed in IR; **enforced in generated Convex code**.

### Convex

Persistence + reactivity. Schema and functions are **generated** — do not hand-edit `convex/schema.ts`, `convex/mutations.ts`, etc.

### Next.js (`apps/app`)

UI, layout, forms, navigation, auth UI. **Not** the domain backend. Frontend calls **Convex client** (target) or temporary shims (`apiFetch` → migrate away).

### `apps/api` (legacy, being removed)

Domain API routes and runtime interpreter are **legacy**. Do not add domain routes. Temporary shims only until frontend uses Convex directly.

## 5. Source-of-truth chain

```txt
.manifest source  →  compile  →  IR  →  Convex projection  →  convex/
```

Drift gates (CI) enforce: committed `convex/*` matches IR regeneration.

Never hand-edit generated Convex files. Fix IR or projection config, then `pnpm manifest:generate-convex`.

---

## 6. Workflow (binding)

```txt
1. Edit manifest/source/*.manifest
2. pnpm manifest:compile
3. pnpm manifest:generate-convex
4. pnpm manifest:check-convex-drift          # CI gate
5. pnpm manifest:enforce-convex-architecture # CI gate
6. npx convex dev                            # push schema, run backend
7. Commit IR + convex/ together
```

Full pipeline: `pnpm manifest:build` (compile + generate-convex + registries).

## 7. CI enforcement

These must pass on every PR:

| Gate | Command |
|------|---------|
| IR compile | `pnpm manifest:compile` |
| Invariants | `pnpm manifest:verify-invariants` |
| IR validate | `pnpm manifest:validate` |
| **Convex drift** | `pnpm manifest:check-convex-drift` |
| **Architecture** | `pnpm manifest:enforce-convex-architecture` |
| Governance registries | compile + registries + git diff |

Prisma schema parity, route drift against Next.js API, and `prisma:check` are **obsolete** in this clone and must not block merges.

---

## 8. Agent rules

1. **Never** propose a store adapter or `RuntimeEngine` in the request path for this repo.
2. **Never** add Prisma models, migrations, or `@repo/database` usage in new code.
3. **Never** hand-write Convex mutations outside the projection.
4. **Never** describe reactions as "Convex schedulers/triggers" — they are **inline transactional fan-out**.
5. Domain change → edit `.manifest` → regenerate → commit IR + `convex/`.
6. UI-only change → `apps/app` only; use Convex queries/mutations from generated API.
7. Legacy code in `manifest/convex-phase-out-registry.json` may exist — **do not extend it**.

---

## 9. When runtime interpreter would return

Only if product requires **live IR evaluation without regeneration** (dynamic tenant rules). That contradicts this clone's drift-gate model. Would require explicit charter amendment and is out of scope for v5.
