# Capsule-Pro Constitution — Manifest → Convex Charter

Version: 2
Status: Binding
Repo: `capsule-pro` @ branch `feat/convex-compile-path-a` (the Convex compile path; `main` is still Prisma)
Supersedes: v1 (which assumed greenfield projection-only and invented parallel scripts)
Audience: Humans, agents, CI, auditors

---

## 0. What this repo is

capsule-pro's governed domain (212 entities, ~1054 commands, in 103 `.manifest` files) is
**compiled to a Convex backend** by the official `ConvexProjection` from `@angriff36/manifest`.
This branch is a **brownfield migration** off Prisma — not greenfield. The generated Convex
backend already exists (`convex/`, ~200 tables, ~1045 mutations, ~600 queries, zero hand-edits);
the work in flight is moving the existing frontend off Prisma onto it (Hermes's 7-phase guide;
currently ~Phase 3).

The single source of truth is the compiled IR (`manifest/ir/kitchen.ir.json`). Everything in
`convex/` and the generated client trio is a **deterministic projection of that IR**, regenerable
and drift-gated.

The proof this works end-to-end is the `convex-example` POC (same 212 entities → 200 tables,
governed writes + reactions + audit, verified on `@angriff36/manifest@2.10.5`).

---

## 1. Authority

Order: **`.manifest` source → compiled IR (`kitchen.ir.json`) → generated `convex/` → generated client trio.**

If hand-edited generated code conflicts with a fresh regen, the hand edit is wrong. If generated
code conflicts with IR, IR wins — fix the source/options and regenerate. Never hand-patch
generated output or route around it.

Projection options are authoritative in **`manifest.config.yaml`** (read via `read-config.mjs`),
not duplicated in scripts: `referenceMode: stringId`, `policyMode: enforce`,
`emitEventsTable: true`, `eventsTable: manifestEvents`. **`stringId` is load-bearing** — the
frontend and generated client use string UUIDs everywhere; do not change it.

---

## 2. The generated artifacts (all KEEP, all regenerated, never hand-edit)

**Backend** — `pnpm manifest:generate-convex` → `convex/`:
`schema.ts · queries.ts · mutations.ts · crons.ts · http.ts · sagas.ts`
Each carries `// GENERATED by the Manifest → Convex projection. DO NOT EDIT.`
The mutations embed governance fail-closed: policies → guards → constraints → mutate →
`manifestEvents` row → inline reactions.

**Frontend trio** — `pnpm manifest:client` → `apps/app/app/lib/`:
- `manifest-types.generated.ts` — entity TS types
- `manifest-client.generated.ts` — async `listX()/getX()` + `executeCommand` (the read/write client)
- `manifest-hooks.generated.ts` — TanStack hooks wrapping the client (currently ~unused; see §4)

---

## 3. The write path (the only governed door)

A governed write is a generated Convex `mutation`. Application code reaches it through:

```
component / action  →  executeCommand("Entity","command", input)   (manifest-client.ts)
                    →  api.mutations.Entity_command  (generated, governed: policies→guards→constraints→mutate→event→reactions, fail-closed)
                    →  Convex persists + reactions fan out inline (one transaction, no outbox)
```

No governed write may bypass a generated mutation. **Forbidden** (enforced by
`enforce-convex-architecture.mjs`): `@prisma/client`, `@repo/database`, `prismaOverride`,
`createManifestRuntime` (the runtime-interpreter path), and hand-written Convex `mutation(`
outside generated `convex/` files.

---

## 4. The read path (and the real-time decision)

Reads are non-authoritative and may go two ways — **this is a deliberate choice per surface:**

| Surface | Use | Real-time? |
|---|---|---|
| Server components / server actions; static/detail/report pages | `await client.listX()` / `getX()` (`manifest-client.generated`) | ❌ one-shot fetch (`client.query()` via read-bridge) |
| Client components that must update live (boards, kanban, dashboards, multi-user) | native `useQuery(api.queries.listX)` from `convex/react` | ✅ live subscription |

**Architectural fact:** `manifest-client.generated` calls `client.query()` — a one-shot await,
not a subscription. It **cannot** be real-time. Real-time exists **only** through native
`convex/react` `useQuery`. The vision's "live by default" is therefore **opt-in per surface**:
put live surfaces on `convex/react`, leave the rest on the async client.

The TanStack `manifest-hooks.generated.ts` is a third option (cached, still not live). It is
currently ~unused; either commit to it for client CRUD or delete it — do not leave it as a
silent third path.

Reads must respect tenant isolation (§6). Reads must not mutate state, define transitions,
synthesize events, or become hidden write paths.

---

## 5. Generation & drift — the lock

`pnpm manifest:ci` is the gate (already wired in `package.json`):
`compile → verify-invariants → validate → generate-convex → check-convex-drift →
enforce-convex-architecture → registries → git diff governance`.

- **IR drift:** committed `manifest/ir/kitchen.ir.json` must equal a recompile of source.
- **Convex drift:** committed `convex/` must equal a fresh `generate-convex` (`check-convex-drift.mjs`).
- **Architecture:** `enforce-convex-architecture.mjs` is baseline-driven — it grandfathers the
  ~289 existing violations (`baselines/convex-architecture.json`) and **fails on new ones**. The
  baseline shrinks as migration proceeds; `:strict` requires zero. CI runs `:strict`.

Bumping `@angriff36/manifest` is a regeneration event: regenerate + commit `convex/` and the
client trio together.

---

## 6. Tenant isolation (security-critical — Convex has no RLS)

1. **Writes:** tenant/org id passed to a generated mutation must be derived from authenticated
   Clerk/Convex context server-side, **never** client-supplied. (`policyMode: enforce` +
   `ctx.auth.getUserIdentity()`.)
2. **Reads:** queries scope by tenant (`withIndex`/`listXBy`). The async client passes `tenantId`
   from server context; native `useQuery` callers must pass the authenticated tenant. An unscoped
   governed read is nonconforming.

This is Hermes's flagged hard part (Phase 5). Until Clerk-Convex auth + tenant filtering are
wired and `policyMode: enforce` is verified live, treat tenant isolation as `MISSING ENFORCEMENT`
and enforce by review.

---

## 7. Forbidden / legacy (the carcass)

Being removed (Hermes Phase 7). Do not add new references; migrate existing ones:

- `@repo/database` / `packages/database/**` (Prisma) — reads move to the generated client.
- `apps/api/**` (Next.js REST dispatcher + read routes) — the `nextjs` projection in
  `manifest.config.yaml` is **legacy**; Convex mutations/queries are the API.
- `manifest/runtime/**` (RuntimeEngine interpreter) — `createManifestRuntime` is forbidden;
  delete after the last app callsites move off it.
- `@prisma/client`, `prisma`, Neon migrations, and the Prisma-derived projections
  (prisma-store/kysely/drizzle/openapi-from-prisma).

`@tanstack/react-query` is **NOT** forbidden — it powers `manifest-hooks.generated`. Keep it if
you adopt the hooks; remove only if you delete that file (§4).

## 8. Change protocol

1. Edit `.manifest` source.
2. `pnpm manifest:compile` (IR).
3. `pnpm manifest:generate-convex` + `pnpm manifest:client` (regenerate backend + trio).
4. `pnpm manifest:ci` (gate).
5. Update read models / UI as needed.

IR-first, always. Never hand-edit generated `convex/` or the trio and retrofit IR later.

Known IR authoring fixes to land (backend-agnostic, from the POC diff): in
`manifest/source/platform/reactions.manifest` — required arrays need `= []` defaults; money/decimal
reaction params need string literals (`"0"`, `"1"`) not numbers.

---

## 9. Agent rules

Classify first: governed write · read · UI · generation · infra/migration · unknown.

- **Governed write** → change `.manifest`, regenerate. Never hand-edit `convex/`.
- **Read** → async client (server) or native `useQuery` (live); keep tenant isolation.
- **UI** → call generated mutations/queries; don't implement rules.
- **Generation** → change source/options, regenerate; keep output deterministic.
- **Migration** → use the existing helpers (`codemod-prisma-to-manifest-client.mjs`,
  `migrate-prisma-imports.mjs`); don't hand-roll new ones.
- **Unknown** → inspect source/IR/`convex/`/`enforce-convex-architecture.mjs`/`manifest.config.yaml`.
  Don't infer semantics from route names. Don't re-decide the architecture — it's decided here.

On conflict, report:

```
NONCONFORMANCE:
- file: / behavior: / violated rule: / safer replacement: / whether patched:
```

Don't preserve nonconforming patterns because they exist. Don't hand-edit generated output to
pass the gate. Don't revive Prisma, `@repo/database`, or `createManifestRuntime`.

---

## 10. Plain terms

Manifest source → IR → Convex backend (generated, governed) + a generated client. Convex runs and
persists it. Clerk says who's acting. The frontend reads via the async client (server) or native
Convex hooks (live), and writes via `executeCommand` → generated mutations.

Only generated Convex mutations govern writes. Reads are flexible. No Prisma, no RuntimeEngine, no
hand-edited generated code. Real-time is opt-in per surface via `convex/react`. If committed
generated code doesn't match a regen, it's drift. If the repo disagrees with this charter, fix the repo.
