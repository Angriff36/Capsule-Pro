## Goal
Implement AND **wire up AND utilize** the genuinely-new 2.0 capabilities so each is
reachable from a real entry point and **proven by an executed runtime roundtrip** ÔÇö not left
as a dangling IR block. Where a feature replaces hand-rolled logic, the hand-rolled path is
retired or redirected through the runtime.

## ÔŤö DEFINITION OF DONE (non-negotiable ÔÇö the anti-dangling gate)
A feature is **NOT done** until ALL of these are true and have **captured evidence**:
1. **Authored** in `manifest/source/*.manifest`.
2. **Compiled** ÔÇö present in `manifest/ir/kitchen.ir.json` (grep-proven: the relevant key is non-empty).
3. **Runtime honors it** ÔÇö the `RuntimeEngine` actually evaluates/executes it (verified by reading
   the runtime path, not assumed).
4. **Reachable** ÔÇö invocable from a real entry point (the dispatcher/route, `runtime.onEvent`, or a
   wired runtime option). NO feature may sit in the IR with nothing calling it.
5. **Proven** ÔÇö an executed test or `.tmp` roundtrip harness shows the behavior end-to-end, with the
   output pasted into `2.0-adoption-notes.md`. "Typecheck passes" and "IR compiles" are necessary but
   **NOT sufficient**.
6. **Hand-code reconciled** ÔÇö any hand-rolled equivalent it supersedes is retired, redirected, or has
   a registry entry explaining why it stays.
7. **No new generated-surface drift**; constitution-compliant (governed writes via runtime).

If any of 1ÔÇô7 is unmet, the phase stays `[ ]` and the gap is logged. Do not claim completion.

## Phase 0 RESULTS (DONE 2026-06-02) ÔÇö see `2.0-adoption-notes.md ┬žE` for evidence
Reading the runtime engine changed the plan. Key findings (the anti-dangling payoff):
- **Reactions auto-dispatch inside `runCommand`** (no host wiring needed) BUT **create-reactions
  silently no-op** (the reaction always sets `instanceId`, which disables autoCreate). The marketed
  "EventCreated Ôćĺ create downstream" pattern does NOT work in 2.0.6. Reactions = transitions only.
- **Sagas are fully viable** (`runSaga`; caller controls per-step instanceId so create+transition both
  work; reverse-order compensation). Best clean pilot.
- **Approvals are in-memory only** (no `approvalStore`, role-as-userId policy hack) Ôćĺ NOT
  production-wireable without a Capsule persistence shim. Authoring an approval block today = dangling.
- Encryption/middleware/profiling/inheritance are independent of these quirks.

**Reordered scope:** Sagas Ôćĺ Reactions(transition) Ôćĺ Encryption/Middleware/Inheritance Ôćĺ Approvals(after shim).
Phase 1 (Approvals-first) is SUPERSEDED ÔÇö moved behind the persistence shim.
