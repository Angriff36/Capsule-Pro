## Part 1 â€” Original vision (handwritten)

This is to be the canonical list of capsule-pro pnpm scripts that i want to exist, defined by their actual real functionality. AI agents MUST ensure that the scripts assigned to this index exist in FULL, with a complete explanation of how exactly they are accomplishing what I am defining here, not just the commands, a full breakdown of ALL the commands and configurations being used to construct the script.

**1. `pnpm manifest:validate`** â€” validate all the manifest soures in `manifest/source/`. This script should check all the sources to find out if they are valid format, ensure they have all the necessary definitions of entities with their full list of advanced functionality and configurations and relationships. It should also ensure that there are no unnecessary or annoying guards in place that do nothing except annoy the user or expose them to internal coding leaks. Users should never be forced to enter or find numbers or fields or anything that manifest itself can provide. This step should also ensure that manifest isn't ever asking for or requiring a field which should be auto generated by manifest, or required when the user cant even have access to it.

**2. `pnpm manifest:compile`** â€” compiles all the `.manifest` files into a single ir AND ensures that the single IR is being consumed everywhere it needs to be. It should normalize all case to the same type (the same type that the db will use) so that no code will ever drift or conflict due to naming, case, types, etc. it should also use similar rules like `packages/typescript-config/base` does, ensuring there aren't things like unused entities or relationships, or things that arent actually wired up properly, or fake implementation, or ai slop.

**3. `pnpm manifest:build`** â€” brings everything together, connecting the code, the db, the ir, the API, hooks, ui elements. It should bridge everything to be interconnected.

The upstream manifest CLI is single-file â†’ single-surface â†’ one artifact per invocation. Capsule-pro is the opposite: 104 `.manifest` files (103 domain + `_base.manifest`) across 13 domain folders, custom projection options, post-generation patching, a typed client the CLI doesn't emit, and CI gates that assert "committed code == freshly generated code." None of that is expressible through `manifest compile` / `manifest generate` alone, so the scripts are the orchestration + glue layer.

### Source of truth


### Schema & DB workflow (when vision requires persisted entities)

1. Edit `manifest/source/<domain>/*.manifest`
2. `pnpm manifest:compile`
3. `pnpm manifest:generate-schema` (alias `manifest:schema:full`)
4. `pnpm manifest:schema:check`
5. Promote IR model blocks into `packages/database/prisma/schema.prisma` (Phase 2b will automate; use `manifest/prisma-options.config.json` â€” never invent `@@map`)
6. `pnpm db:dev -- --create-only --name <name>` â€” review SQL
7. `pnpm db:deploy`
8. `pnpm db:check`

If `db:check` fails after steps 1â€“8, the failure is procedural â€” not "normal drift."

### What Capsule scripts do that the upstream CLI cannot

- **Multi-module merge** â€” `compile.mjs` + `compileProjectToIR` (CLI `--glob` is broken for 104 modules)
- **Domain route remapping** â€” `generate.mjs` + `ENTITY_DOMAIN_MAP`
- **Post-generation patching** â€” accessor config, field overrides, orphan cleanup
- **Typed Capsule client** â€” `generate-capsule-client.mjs`
- **CI drift gates** â€” byte-compare committed artifacts vs fresh regeneration (`schema:check`, `openapi:check`, `react-query:check`, `ir:embed:check`, `verify-invariants`, governance registry diff)
