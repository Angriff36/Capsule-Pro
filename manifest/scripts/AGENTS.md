## 📄 Manifest doc — `integration/prisma` · Prisma schema projection

The **Prisma projection** reads compiled Manifest IR and emits a `schema.prisma` artifact. It is compile-time only — it does not connect to a database, run migrations, or know anything about any specific application. All relational concepts (table names, column names, decimal precision, composite indexes, foreign-key field names) live in projection config; Manifest core stays backend-neutral.

The projection ships **inside the main `@angriff36/manifest` package** — no separate install needed. Running `manifest generate -p prisma` works out of the box.

<Note>
An earlier pre-release explored shipping the projection as `@manifest/projection-prisma`. That package was never published. Since v1.0.0 the projection is bundled in `@angriff36/manifest` with the same internal separation (`src/manifest/projections/prisma/`) as the Next.js and routes projections.
</Note>

For programmatic use, import from the main package:

```ts
import { PrismaProjection } from '@angriff36/manifest/projections/prisma';
```

## Quickstart

A minimal `.manifest` source:

```manifest widget.manifest
entity Widget {
  property required id: string
  property required name: string
  property required qty: int
  property required unitCost: money
}

store Widget in durable
```

A minimal `manifest.config.yaml`:

```yaml manifest.config.yaml
projections:
  prisma:
    output: generated/
    options:
      provider: postgresql
```

Compile and generate:

```bash
manifest compile widget.manifest
manifest generate ir/ -p prisma -s prisma.schema -o generated/
```

Emitted `generated/schema.prisma`:

```prisma
// Auto-generated by @angriff36/manifest
// DO NOT EDIT — regenerate with the projection.

datasource db {
  provider = "postgresql"
}

generator client {
  provider = "prisma-client-js"
}

model Widget {
  id String @id
  name String
  qty Int
  unitCost Decimal @db.Decimal(12, 2)
}
```

Emitted `generated/prisma.config.ts` (Prisma 7+ — URL lives here, not in `schema.prisma`):

```typescript
// Auto-generated by @angriff36/manifest
// Prisma 7+: connection URL lives here, not in schema.prisma.
import { defineConfig } from 'prisma/config';

export default defineConfig({
  datasources: { db: { url: process.env.DATABASE_URL } },
});
```

## What gets emitted, what gets skipped

| IR shape | Projection behavior |
| --- | --- |
| `entity X { ... }` + `store X in durable` (or `postgres` / `supabase`) | Emits a Prisma `model` |
| `entity X { ... }` + `store X in memory` (or `localStorage`) | **Skipped** with `PRISMA_SKIPPED_NON_DURABLE` info |
| `entity X { ... }` with no `store` declaration | **Skipped** with `PRISMA_SKIPPED_NO_STORE` info |
| `external entity X { ... }` | **Skipped** with `PRISMA_SKIPPED_EXTERNAL` info |
| `property` entries | Emitted as Prisma fields |
| `computed` entries | **Never emitted** — derived values must not become columns |

Entities are iterated in IR source order; the projection does not re-sort.

<Warning>
A property literally named `id` becomes the `@id` field. Entities without an `id` property get an info diagnostic (`PRISMA_NO_ID_PROPERTY`) because Prisma's schema validator will reject a model with no primary key.
</Warning>

## Type mapping

The IR carries `type.name` as an arbitrary string (Manifest's type vocabulary is open). The projection holds the mapping table:

| IR `type.name` | Prisma scalar |
| --- | --- |
| `string`, `text`, `uuid` | `String` |
| `boolean`, `bool` | `Boolean` |
| `int` | `Int` |
| `bigint` | `BigInt` |
| `float` | `Float` |
| `decimal`, `money` | `Decimal` (with default `@db.Decimal(12, 2)`) |
| `date`, `datetime` | `DateTime` |
| `json` | `Json` |
| `bytes` | `Bytes` |
| anything else | **diagnostic** (`PRISMA_UNKNOWN_TYPE`) |

The default mapping does **not** include `number`. That is deliberate.

## The `number` ambiguity rule

Manifest's `number` type does not distinguish integers from real numbers from money. Silently mapping it to `Float` is the class of silent rounding bug this project exists to prevent — especially for currency. So the projection refuses to guess:

```manifest
entity Order {
  property required id: string
  property required total: number  // ← ambiguous
}
```

The projection emits an error diagnostic:

```
PRISMA_AMBIGUOUS_NUMBER (Order):
  Property 'Order.total' is typed 'number', which is ambiguous
  (Manifest does not distinguish integers from real numbers from money).
  Pick a precise type in the .manifest source: 'int' or 'bigint' for
  counts and ids, 'float' for measurements where rounding is acceptable,
  'money' or 'decimal' for currency and other exact-decimal values.
  Or supply a 'typeMappings.Order.total' override.
```

The ambiguous column is **skipped** in the output. Fix the source:

```manifest
entity Order {
  property required id: string
  property required total: money  // ← exact-decimal, picks up @db.Decimal(12, 2)
}
```

<Note>
The escape hatch is the projection config's `typeMappings` — supply an explicit Prisma scalar there and the diagnostic disappears. Useful when migrating a legacy `.manifest` file that already used `number` for a field whose precision intent is recorded elsewhere.
</Note>

## Money and decimal precision

`money` and `decimal` both resolve to Prisma `Decimal`. Either gets a default `@db.Decimal(12, 2)` — the conservative money default (up to 9,999,999,999.99 with cent precision).

```manifest
entity Order {
  property required total: money       // → Decimal @db.Decimal(12, 2)
  property required fxRate: decimal    // → Decimal @db.Decimal(12, 2)
}
```

Override per-property via `precision`:

```yaml
projections:
  prisma:
    options:
      precision:
        Order:
          fxRate: { precision: 18, scale: 8 }   # → @db.Decimal(18, 8)
```

The rule is keyed on the **resolved Prisma scalar**, not the IR type name. A property routed to `Decimal` via `typeMappings` picks up the same default precision.

## Relationships

All four IR relationship kinds emit real Prisma `@relation`:

| IR | Emitted Prisma | Notes |
| --- | --- | --- |
| `hasMany name: T` | `name T[]` | Parent side of 1:N |
| `hasOne name: T` | `name T?` | Parent side of 1:1 |
| `belongsTo name: T` | `nameId String` + `name T @relation(fields: [nameId], references: [id])` | Child side; FK type follows target's `id` |
| `ref name: T` | same shape as `belongsTo` | "Loose" — no missing-backside warning |

### 1:1 vs 1:N

The projection detects the cardinality from the opposite side:

```manifest
entity User {
  property required id: string
  hasOne profile: Profile        // 1:1 — declares opposite as hasOne
}

entity Profile {
  property required id: string
  belongsTo user: User           // child side → FK gets @unique
}
```

Emits:

```prisma
model User {
  id String @id
  profile Profile?
}

model Profile {
  id String @id
  userId String @unique          // ← @unique because opposite is hasOne
  user User @relation(fields: [userId], references: [id])
}
```

For 1:N (parent side is `hasMany`), the FK is **not** `@unique`.

### Foreign key field name

Default is `${relationshipName}Id` (camelCase). Override at IR level via the `foreignKey` annotation, or at projection-config level via `foreignKeys`:

```yaml
projections:
  prisma:
    options:
      foreignKeys:
        Book:
          author: writerId       # → Book.writerId instead of Book.authorId
```

The FK type follows the target entity's `id` property type. A `string` PK gets a `String` FK; an `int` PK gets an `Int` FK. No type mismatches.

### Foreign key columns honor `columnMappings`

FK fields are virtual properties from the consumer's perspective; the existing `columnMappings` knob applies:

```yaml
projections:
  prisma:
    options:
      columnMappings:
        Book:
          authorId: author_id    # → authorId String @map("author_id")
```

### `belongsTo` alongside an explicit FK property

If your `.manifest` source declares the FK property explicitly **and** declares the relationship, the projection deduplicates — it does NOT emit a second column with the same name:

```manifest
entity Book {
  property required id: string
  property required authorId: string    // ← explicit FK column
  belongsTo author: Author              // ← relationship using that column
}
```

Emits:

```prisma
model Book {
  id String @id
  authorId String                        // ← from the property declaration
  author Author @relation(fields: [authorId], references: [id])
}
```

The relation step sees that `authorId` already exists as a property and reuses it. The FK column line is **not** re-emitted; the `@relation(fields: [authorId], ...)` clause still wires it.

If you supply both an explicit FK property **and** a `foreignKeys` config override that points the relationship at a different column name, the override wins for the relation field — but the explicit property still emits as its own column, so you'll end up with both columns in the model. Pick one approach per relationship.

### Self-relations

A one-sided self-relation triggers `PRISMA_RELATION_MISSING_BACKSIDE`:

```manifest
entity Tree {
  property required id: string
  belongsTo parent: Tree   // no `children` declared → warning
}
```

A two-sided self-relation (`parent` + `children` on the same entity) is the textbook case where Prisma requires `@relation("name")` to disambiguate which `belongsTo` matches which `hasMany`. The projection refuses to invent names and emits `PRISMA_RELATION_AMBIGUOUS` for both sides — hand-add the relation names after generation, or refactor to a single relation:

```prisma
model Tree {
  id String @id
  parentId String?
  parent Tree? @relation("TreeHierarchy", fields: [parentId], references: [id])
  children Tree[] @relation("TreeHierarchy")
}
```

### MongoDB provider

When `provider: mongodb`, Prisma's Mongo connector requires the `@id` column to map to `_id` with ObjectId semantics. The projection applies the canonical pattern automatically:

```yaml
projections:
  prisma:
    options:
      provider: mongodb
```

```manifest
entity Order {
  property required id: string
  property required total: money
}
store Order in durable
```

Emits:

```prisma
model Order {
  id String @id @default(auto()) @map("_id") @db.ObjectId
  total Decimal @db.Decimal(12, 2)
}
```

The Mongo-specific attributes are auto-applied only when you have NOT supplied your own `columnMappings.<Entity>.id` or a literal `defaultValue` on the `id` property. If you want a non-ObjectId pattern (e.g. plain string ULIDs), override via `typeMappings.<Entity>.id` to something other than `String` — the projection will skip the ObjectId attribute but still emit `@map("_id")`.

## Configuration options

All per-property options use the nested `Record<EntityName, Record<PropertyName, ...>>` shape. There are no flattened `"Entity.property"` dotted-string keys anywhere in the config.

```yaml
projections:
  prisma:
    # Path hint for the emitted artifact (default: schema.prisma).
    output: generated/schema.prisma

    options:
      # Optional Prisma datasource provider. When set, the projection
      # emits a `datasource` block at the top of the artifact.
      provider: postgresql       # one of: postgresql | mysql | sqlite | sqlserver | mongodb | cockroachdb

      # Per-entity table-name override → @@map("...").
      tableMappings:
        Order: orders
        Customer: customers

      # Per-entity, per-property column-name override → @map("...").
      columnMappings:
        Order:
          createdAt: created_at
          updatedAt: updated_at

      # Automatic identifier casing (opt-in). Emits @map/@@map for any name
      # not already in the target case, so you don't hand-write the mappings
      # above for every field. Explicit tableMappings/columnMappings still win.
      naming: snake_case          # shorthand: snake_case columns + pluralized snake_case tables
      # Object form:
      # naming:
      #   table: snake_case        # snake_case | camelCase | PascalCase | preserve
      #   column: snake_case       # snake_case | camelCase | preserve
      #   pluralizeTables: true    # Widget → widgets (default true)

      # Per-entity, per-property decimal precision/scale.
      # Applies to any property whose resolved scalar is Decimal.
      precision:
        Order:
          total: { precision: 14, scale: 2 }
          fxRate: { precision: 18, scale: 8 }

      # Per-entity composite/named index definitions → @@index([...]).
      indexes:
        Order:
          - ["customerId", "createdAt"]
          - { fields: ["status"], name: "order_status_idx" }

      # Per-entity, per-property type override. Bypasses the default
      # IR-type-name → Prisma-scalar mapping. Value is the literal
      # Prisma scalar (e.g. "Int", "BigInt", "Decimal", "DateTime").
      typeMappings:
        Order:
          legacyAmount: Decimal   # opt out of PRISMA_AMBIGUOUS_NUMBER for a `number` field

      # Per-entity, per-relationship FK field name override.
      # Overrides default `${relationshipName}Id` and IR `foreignKey` annotation.
      # Value is a plain string (field name) or a structured object with referential actions.
      foreignKeys:
        Order:
          customer: customerRef   # → Order.customerRef instead of Order.customerId
          # Structured form — includes referential actions:
          # customer:
          #   fields: [customerId]
          #   references: [id]
          #   onDelete: Cascade
          #   onUpdate: NoAction

      # Per-entity, per-property native @db.* attribute (without the @db. prefix).
      # Applied after the Prisma scalar but before @default.
      # Use for types the projection can't infer, or to override the default @db.Decimal(12,2).
      dbAttributes:
        Order:
          amount: Decimal(14, 2)     # → @db.Decimal(14, 2)
          externalId: VarChar(36)    # → @db.VarChar(36)

      # Per-entity, per-property array of verbatim Prisma field attributes.
      # Appended after type and dbAttributes. Use for @unique, @updatedAt, @default(...), etc.
      fieldAttributes:
        Order:
          code: ["@unique"]
          updatedAt: ["@updatedAt", "@default(now())"]

      # Environment variable name for the database URL in the emitted prisma.config.ts.
      # Default: DATABASE_URL. Only relevant when `provider` is set.
      urlEnvVar: POSTGRES_CONNECTION_STRING
```

## Automatic casing (the `naming` convention)

By default the projection emits IR names verbatim — to render camelCase IR identifiers as `snake_case` database names you'd hand-write a `columnMappings` entry per field. The `naming` option automates that:

```yaml
projections:
  prisma:
    options:
      naming: snake_case
```

`createdAt` emits `@map("created_at")`, `Widget` emits `@@map("widgets")`. The shorthand `snake_case` expands to `{ table: snake_case, column: snake_case, pluralizeTables: true }`; the object form tunes each axis independently.

**The convention only ever adds `@map`/`@@map`.** The Prisma *model name* and *field identifiers* stay as the IR name, so relation `fields`/`references`, `@@id`, `@@unique`, and `@@index` are unaffected — only the physical database name changes, and a `@map`/`@@map` is emitted only when the physical name actually differs (so `id` stays bare):

```prisma
model UserAccount {
  id          String   @id
  createdAt   DateTime @map("created_at")
  displayName String   @map("display_name")

  @@map("user_accounts")
}
```

Resolution order per name:

1. explicit `tableMappings` / `columnMappings` (always win — also the **escape hatch** for irregular plurals the built-in pluralizer gets wrong)
2. the `naming` convention
3. the IR name verbatim

A **global** `naming` can also be set once at the top level of `manifest.config.yaml` and is inherited by the Prisma projection; a per-projection `projections.prisma.options.naming` overrides the global default.

## `prisma.config.ts` companion

When `provider` is set, the projection emits **two artifacts** from a single `generate()` call:

| Index | Path hint | Content |
| --- | --- | --- |
| `artifacts[0]` | `schema.prisma` | The full Prisma schema |
| `artifacts[1]` | `prisma.config.ts` | Prisma 7+ config file with `url: process.env.<urlEnvVar>` |

Prisma 7 moved the connection URL out of `schema.prisma` into a `prisma.config.ts` file. The projection handles this automatically so `prisma migrate` works without edits:

```typescript
// prisma.config.ts — emitted automatically when provider is set
import { defineConfig } from 'prisma/config';

export default defineConfig({
  datasources: { db: { url: process.env.DATABASE_URL } },
});
```

Customize the env var name via `urlEnvVar`:

```yaml
projections:
  prisma:
    options:
      provider: postgresql
      urlEnvVar: POSTGRES_CONNECTION_STRING
```

Without `provider`, only `schema.prisma` is emitted (no datasource block, no `prisma.config.ts`). Use this when you manage datasource configuration externally.

## Diagnostics reference

The projection emits structured diagnostics rather than silently doing the wrong thing. Each carries an entity name (where applicable) and a resolution hint. The CLI's `manifest generate` command **exits non-zero** when any error-severity diagnostic is returned, so CI pipelines detect generation failures.

| Code | Severity | Trigger | Resolution |
| --- | --- | --- | --- |
| `PRISMA_AMBIGUOUS_NUMBER` | error | Property typed `number` with no `typeMappings` override | Pick `int`/`bigint`/`float`/`money`/`decimal` in source, or supply `typeMappings.<Entity>.<Property>` |
| `PRISMA_UNKNOWN_TYPE` | error | `type.name` not in default mapping and not in `typeMappings` | Add a `typeMappings` entry or change the source type |
| `PRISMA_NO_ID_PROPERTY` | info | Entity has no property named `id` | Add `property required id: string` |
| `PRISMA_SKIPPED_EXTERNAL` | info | Entity marked `external entity` | Expected — projection intentionally skips external entities |
| `PRISMA_SKIPPED_NO_STORE` | info | Entity has no `store` declaration | Add `store EntityName in durable` to opt in |
| `PRISMA_SKIPPED_NON_DURABLE` | info | Store target is `memory` or `localStorage` | Flip to `durable` (or `postgres`/`supabase`) to opt in |
| `PRISMA_RELATION_VIA_THROUGH_UNIMPLEMENTED` | info | `hasMany ... through X` (explicit M2M) | Declare the join entity (`X`) with its own belongsTo relations |
| `PRISMA_RELATION_AMBIGUOUS` | info | Multiple relationships between same entity pair (including two-sided self-relations like `parent` + `children` on the same type) | Refactor to a single relation, or hand-edit `@relation("name")` after generation |
| `PRISMA_RELATION_MISSING_BACKSIDE` | warning | One-sided `hasMany` / `hasOne` / `belongsTo` / `ref` — all four emit a full `@relation` that Prisma rejects without the opposite side | Declare the matching relation on the other entity |
| `PRISMA_RELATION_TARGET_NOT_EMITTED` | warning | Relationship targets an entity that is not emitted as a Prisma model (external entity, memory/localStorage store, no store) | Declare the target as a durable entity, or remove the relationship |

## What the projection deliberately does NOT do

These are not bugs — they're scope boundaries:

- **No database connection.** The projection is pure IR → text. Running migrations is your job (or Prisma's).
- **No `@relation("name")` auto-generation.** Multi-relation pairs require named `@relation` clauses; the projection refuses to invent names. Diagnose, don't guess.
- **No join-entity inference for `through` relationships.** Manifest's `through` annotation declares the join model, but Prisma needs the join entity to be a fully-modeled entity with its own belongsTo declarations. The user provides those.
- **No `id` modifier.** Primary-key designation is by literal property name (`id`). A dedicated `@id` modifier in core grammar is a possible future addition; the projection will pick it up when it lands.
- **No app coupling.** No defaults reference tenant columns, soft-delete columns, auth fields, or any other application-specific concept.

## CLI usage

The projection registers automatically; `manifest generate -p prisma` works out of the box:

```bash
# Single source file
manifest compile orders.manifest
manifest generate ir/ -p prisma -s prisma.schema -o generated/

# One-step build
manifest build orders.manifest -p prisma -s prisma.schema --code-output generated/
```

The CLI's `--projection` flag accepts `prisma` alongside `nextjs`. The surface name `prisma.schema` is the projection's single emission surface.

## Programmatic usage

For build scripts that need direct control:

```typescript
import { PrismaProjection } from '@angriff36/manifest/projections/prisma';
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import fs from 'fs/promises';

const source = await fs.readFile('orders.manifest', 'utf8');
const { ir, diagnostics: compileDiags } = await compileToIR(source);

if (!ir) {
  console.error('Compilation failed:', compileDiags);
  process.exit(1);
}

const projection = new PrismaProjection();
const result = projection.generate(ir, {
  surface: 'prisma.schema',
  options: {
    provider: 'postgresql',
    tableMappings: { Order: 'orders' },
    precision: { Order: { total: { precision: 14, scale: 2 } } },
  },
});

for (const d of result.diagnostics) {
  console[d.severity === 'error' ? 'error' : 'log'](`[${d.code}] ${d.message}`);
}

const errors = result.diagnostics.filter((d) => d.severity === 'error');
if (errors.length > 0) process.exit(1);

for (const artifact of result.artifacts) {
  await fs.writeFile(artifact.pathHint!, artifact.code, 'utf8');
}
```

## 📄 Manifest doc — `integration/nextjs` · Use Manifest with Next.js App Router

This guide walks you through adding Manifest to an existing Next.js App Router project. By the end, you will have a `.manifest` domain model, generated route handlers for reads and writes, and Clerk auth wired into the identity pipeline.

## Prerequisites

- Next.js 13+ with the App Router
- Node.js 18+
- An existing Next.js project (or `npx create-next-app@latest`)

## Workflow

<Steps>
  <Step title="Install the package">
    Add `@angriff36/manifest` to your project:

    ```bash
    npm install @angriff36/manifest
    ```
  </Step>

  <Step title="Write your .manifest file">
    Create a file describing your domain model. This example models a `Recipe` entity with a `publish` command:

    ```manifest
    entity Recipe {
      property required id: string
      property required name: string
      property category: string?
      property status: string = "draft"
      property authorId: string

      command publish() {
        guard self.status == "draft"
        guard user.id == self.authorId
        mutate status = "published"
        emit RecipePublished
      }

      policy canPublish execute: user.role in ["author", "editor"]
    }

    store Recipe in memory

    event RecipePublished: "kitchen.recipe.published" {
      recipeId: string
      authorId: string
    }
    ```

    Save this as `recipe.manifest` in your project root.
  </Step>

  <Step title="Generate routes">
    Use the projection CLI to generate the route files. Run these commands from your project root:

    ```bash
    # Compile the .manifest file to IR
    manifest compile recipe.manifest --output ir/

    # Generate all surfaces from the compiled IR
    manifest generate ir/recipe.ir.json \
      --auth clerk \
      --output app/api/

    # Or generate specific surfaces individually
    manifest generate ir/recipe.ir.json --surface types --output lib/manifest-types.ts
    manifest generate ir/recipe.ir.json --surface client --output lib/manifest-client.ts
    ```

    <Note>
      The CLI is a development scaffolding tool. For automated build pipelines, call the projection API programmatically in a build script instead.
    </Note>
  </Step>

  <Step title="Review the generated read route">
    The generated GET handler uses a direct database query — it does not call `RuntimeEngine`. This is intentional: read policies are not enforced at the query level by default, and direct queries are more efficient.

    ```typescript
    // app/api/recipes/route.ts — auto-generated, illustrative
    import { NextRequest } from "next/server";
    import { auth } from "@clerk/nextjs/server";
    import { database } from "@/lib/database";
    import { manifestSuccessResponse, manifestErrorResponse } from "@/lib/manifest-response";

    export async function GET(request: NextRequest) {
      try {
        const { userId } = await auth();
        if (!userId) {
          return manifestErrorResponse("Unauthorized", 401);
        }

        const userMapping = await database.userTenantMapping.findUnique({
          where: { userId },
        });

        if (!userMapping) {
          return manifestErrorResponse("User not mapped to tenant", 400);
        }

        const { tenantId } = userMapping;

        // Direct Prisma query — bypasses runtime deliberately
        const recipes = await database.recipe.findMany({
          where: { tenantId, deletedAt: null },
          orderBy: { createdAt: "desc" },
        });

        return manifestSuccessResponse({ recipes });
      } catch (error) {
        console.error("Error fetching recipes:", error);
        return manifestErrorResponse("Internal server error", 500);
      }
    }
    ```
  </Step>

  <Step title="Review the generated command route">
    The generated POST handler **must** call `RuntimeEngine.runCommand`. Guards, policies, constraints, and event emissions only run through the runtime.

    The generated handler strips client-supplied identity fields (`id`, `userId`, `tenantId`, `orgId`, `user`) from the request body and injects authoritative values from Clerk and the URL path before calling the runtime:

    ```typescript
    // app/api/recipes/commands/publish/route.ts — auto-generated, illustrative
    import { NextRequest } from "next/server";
    import { auth } from "@clerk/nextjs/server";
    import { RuntimeEngine } from "@angriff36/manifest";
    import { ir } from "@/lib/manifest-ir";
    import { manifestSuccessResponse, manifestErrorResponse } from "@/lib/manifest-response";

    export async function POST(
      request: NextRequest,
      { params }: { params: { id: string } }
    ) {
      const { userId } = await auth();
      if (!userId) {
        return manifestErrorResponse("Unauthorized", 401);
      }

      const body = await request.json();

      // Strip client-supplied identity fields — never trust these
      const {
        id: _id,
        userId: _userId,
        tenantId: _tenantId,
        orgId: _orgId,
        user: _user,
        ...safeInput
      } = body;

      const runtime = new RuntimeEngine(ir, {
        user: { id: userId, role: 'author' },
      });

      const result = await runtime.runCommand(
        'publish',
        { ...safeInput },
        { entityName: 'Recipe', instanceId: params.id }
      );

      if (!result.success) {
        return manifestErrorResponse(result.error ?? 'Command failed', 422);
      }

      return manifestSuccessResponse({ events: result.events });
    }
    ```
  </Step>

  <Step title="Wire up Clerk auth">
    If you chose `--auth clerk`, install the Clerk SDK and configure your middleware:

    ```bash
    npm install @clerk/nextjs
    ```

    Add `clerkMiddleware` to `middleware.ts`:

    ```typescript
    import { clerkMiddleware } from '@clerk/nextjs/server';

    export default clerkMiddleware();

    export const config = {
      matcher: ['/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)', '/(api|trpc)(.*)'],
    };
    ```

    Set your environment variables in `.env.local`:

    ```bash
    NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_...
    CLERK_SECRET_KEY=sk_...
    ```

    The generated routes call `auth()` from `@clerk/nextjs/server` to resolve `userId`. No additional configuration is needed in the route files.
  </Step>
</Steps>

## Multi-tenant setup

To enable per-tenant data isolation, keep a `userTenantMapping` table that maps `userId` to `tenantId`. The generated read routes look up this mapping before querying:

```typescript
const userMapping = await database.userTenantMapping.findUnique({
  where: { userId },
});

if (!userMapping) {
  return manifestErrorResponse("User not mapped to tenant", 400);
}

const { tenantId } = userMapping;

const recipes = await database.recipe.findMany({
  where: { tenantId, deletedAt: null },
});
```

Pass `tenantId` to the projection generator to enable filtering:

```bash
manifest generate ir/recipe.ir.json --auth clerk --output app/api/
# Default: includeTenantFilter=true, tenantIdProperty='tenantId'
```

To use a different property name (e.g., `orgId`):

```typescript
// In a build script using the programmatic API
const result = projection.generateRoute(ir, 'Recipe', {
  authProvider: 'clerk',
  tenantIdProperty: 'orgId',
});
```

## Workspace / monorepo setup

In a monorepo, it is common to compile the IR once and share it across packages. A typical layout:

```
apps/
  web/           # Next.js app — consumes generated routes and client SDK
packages/
  domain/        # recipe.manifest, build script, generated IR artifact
  api-types/     # generated ts.types output, shared across apps
```

In `packages/domain`, add a build script that compiles the IR and runs projections:

```typescript
// packages/domain/scripts/build.ts
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import { NextJsProjection } from '@angriff36/manifest/projections/nextjs';
import fs from 'fs/promises';
import path from 'path';

const source = await fs.readFile('recipe.manifest', 'utf-8');
const { ir } = await compileToIR(source);
if (!ir) process.exit(1);

// Write IR artifact for apps to import
await fs.writeFile('dist/recipe.ir.json', JSON.stringify(ir, null, 2));

const projection = new NextJsProjection();

// Write types to the shared types package
const typesResult = projection.generateTypes(ir);
await fs.writeFile(
  path.resolve('../../packages/api-types/src/recipe.ts'),
  typesResult.code
);

// Write routes into the Next.js app
const routeResult = projection.generateRoute(ir, 'Recipe', { authProvider: 'clerk' });
await fs.writeFile(
  path.resolve('../../apps/web/app/api/recipes/route.ts'),
  routeResult.code
);
```

In `apps/web`, import the compiled IR directly:

```typescript
// apps/web/lib/manifest-ir.ts
import ir from '@repo/domain/dist/recipe.ir.json';
export { ir };
```

## Listening to events in route handlers

Register an `onEvent` listener before calling `runCommand` to tap into emitted events. This is useful for publishing to an outbox or triggering webhooks:

```typescript
const runtime = new RuntimeEngine(ir, {
  user: { id: userId, role: 'author' },
});

const unsubscribe = runtime.onEvent((event) => {
  console.log(event.name, event.channel, event.payload);
  // Enqueue to your outbox, publish to Pub/Sub, etc.
});

try {
  const result = await runtime.runCommand(
    'publish',
    {},
    { entityName: 'Recipe', instanceId: params.id }
  );

  if (!result.success) {
    return manifestErrorResponse(result.error ?? 'Command failed', 422);
  }

  return manifestSuccessResponse({ events: result.events });
} finally {
  unsubscribe();
}
```

## Realtime subscriptions (SSE)

Entities flagged with the `realtime` modifier unlock three additional surfaces: an SSE subscription route (`nextjs.subscribe`), a typed auto-reconnecting React hook (`nextjs.subscriptionHook`), and a shared singleton runtime accessor (`nextjs.sharedRuntime`) that lets command routes and subscriptions observe the same event stream:

```manifest
entity Recipe {
  realtime
  property required id: string
  // ...
}
```

```typescript
const projection = new NextJsProjection();
const sseRoute = projection.generate(ir, { surface: 'nextjs.subscribe', entity: 'Recipe' });
const hook = projection.generate(ir, { surface: 'nextjs.subscriptionHook', entity: 'Recipe' });
const shared = projection.generate(ir, { surface: 'nextjs.sharedRuntime' });
```

When any entity is realtime, generated command routes (inline mode) automatically use the shared runtime via `getSharedRuntime()` instead of constructing one per request, so subscribers see events emitted by commands. This requires a long-lived, single-instance server. See [Realtime Subscriptions](/extensibility/realtime-subscriptions) for the full contract.

## Common issues

<AccordionGroup>
  <Accordion title='"Entity not found in IR"'>
    Entity names are case-sensitive. Verify that the name you pass to `runCommand` exactly matches the entity name declared in your `.manifest` file.
  </Accordion>

  <Accordion title="Guard fails even with the correct user">
    Guards reference `user.*` from the runtime context, not from the request body. Make sure you pass `user: { id: userId, role: '...' }` when constructing `RuntimeEngine`. If the guard references `user.role` and you did not supply it, the failure is correct behavior.
  </Accordion>

  <Accordion title="Generated route imports are wrong">
    The generated code uses import paths like `@/lib/database` and `@/lib/manifest-response`. These are conventions — create those modules in your project or pass custom `databaseImportPath` and `responseImportPath` options to the projection generator.
  </Accordion>

  <Accordion title="Tenant filtering returns no results">
    Confirm that your database rows have a `tenantId` column populated and that the `userTenantMapping` lookup is returning the correct value. You can disable tenant filtering during development by passing `includeTenantFilter: false` to the generator.
  </Accordion>
</AccordionGroup>

## 📄 Manifest doc — `projections/react-query` · React Query hooks projection

The React Query projection generates typed [TanStack Query](https://tanstack.com/query) hooks and a `QueryClient` provider from compiled Manifest IR. Reach for it when you are building a React front end against the [Next.js](/integration/nextjs), [Express](/projections/express), or [Hono](/projections/hono) API surfaces and want deterministic, typed data-fetching hooks with automatic cache invalidation rather than hand-written `useQuery`/`useMutation` calls.

The projection is registered under the name `react-query` and lives at `src/manifest/projections/react-query/generator.ts`.

## What it generates

The projection exposes two surfaces.

`react-query.hooks` emits a single self-contained TypeScript module (path hint `src/hooks/manifest-hooks.ts`). It inlines an `interface` for every entity in the IR, an input interface for each command that has parameters, a `queryKeys` factory object, an internal `apiFetch<T>` helper, one list hook and one detail hook per entity, and one mutation hook per entity-scoped command. List hooks (`use<Entity>List`) wrap `GET {apiBasePath}/{entity}/list`; detail hooks (`use<Entity>Detail`) wrap `GET {apiBasePath}/{entity}/{id}` and are gated on a truthy `id`. Mutation hooks (`use<Entity><Command>`) issue `POST {dispatcherBasePath}/{entity}/commands/{command-kebab}` and, on success, call `queryClient.invalidateQueries` for the parent entity before delegating to any caller-supplied `onSuccess`.

`react-query.provider` emits a `'use client'` component module (path hint `src/providers/manifest-query-provider.tsx`) exporting `ManifestQueryProvider`. It constructs a `QueryClient` with `staleTime` set from options, `refetchOnWindowFocus: false`, and — when `errorBoundaryIntegration` is enabled — `throwOnError: true` on both queries and mutations.

A command with no associated entity is skipped and reported with an `ORPHAN_COMMAND` warning diagnostic. An unrecognized surface returns an `UNKNOWN_SURFACE` error diagnostic.

## Usage

```ts
import { getProjection } from '@angriff36/manifest/projections';

const projection = getProjection('react-query');
const result = projection.generate(ir, {
  surface: 'react-query.hooks',
  options: { apiBasePath: '/api', defaultStaleTime: 60_000 },
});

for (const artifact of result.artifacts) {
  console.log(artifact.pathHint, artifact.code);
}
```

Swap the surface to `react-query.provider` for the `QueryClient` setup module.

## Type mapping & behavior

IR property and parameter types are mapped to TypeScript by `irTypeToTsType`: `string` to `string`, `number` to `number`, `boolean` to `boolean`, `date`/`datetime` to `Date`, `any` to `unknown`, and `void` to `void`. Any other IR type name passes through unchanged. A nullable IR type yields `T | null`.

Entity properties are marked optional (`?`) when the property carries the `optional` modifier, has a default value, or is nullable. Command input fields are optional when the parameter is not required, in which case the TypeScript type also gains ` | undefined`. Query keys use the lower-camel-cased entity name (for example `queryKeys.userProfile.all`). The mutation return type is taken from the command's `returns` type, defaulting to `unknown`.

## Options

`ReactQueryProjectionOptions` accepts `apiBasePath` (default `/api`), `dispatcherBasePath` (default `/api/manifest`), `optimisticUpdates` (default `true`), `errorBoundaryIntegration` (default `true`), `typesImportPath` (default `@/types/manifest-generated`), and `defaultStaleTime` in milliseconds (default `30000`). All are optional.

## Notes & limitations

<Warning>
Two declared options are currently inert. `optimisticUpdates` is accepted and normalized but is never read during generation — the generated mutation hooks use a cache-invalidation strategy (`invalidateQueries`) only, with no optimistic update helpers emitted, despite the module's header comment mentioning them. `typesImportPath` is likewise accepted but unused; entity and command-input types are inlined directly into the hooks file rather than imported, so the hooks module is fully self-contained.
</Warning>

There is no dedicated CLI command for this projection. The bundled `manifest generate` command currently wires up the Next.js projection only; to emit React Query hooks, invoke the projection programmatically through `getProjection('react-query')` as shown above.

## 📄 Manifest doc — `projections/additional-projections` · Additional projections

Beyond the primary framework and ORM projections, Manifest ships several specialized projections for infrastructure, analytics, search, and developer tooling targets. Each follows the same `ProjectionTarget` interface and deterministic output contract.

### Kysely

The Kysely projection generates TypeScript type definitions for the [Kysely](https://kysely.dev) type-safe SQL query builder. It produces per-table TypeScript interfaces (e.g., `TaskTable`), a `Database` interface mapping table names to those interfaces, and a `createDb()` factory function for creating configured `Kysely<Database>` instances.

The projection is registered under the name `kysely` and emits one surface: `kysely.schema` (content type `typescript`, default output `schema.ts`).

**Key characteristics:**

- IR types map to Kysely/TypeScript types with `Generated<T>` for ID columns and columns with `defaultValue`, and `ColumnType<SelectType, InsertType, UpdateType>` for Date/datetime columns.
- `belongsTo`/`ref` relationships auto-generate FK columns (skipped if an FK property already exists). `hasMany`/`hasOne` are expressed through Kysely's relation API, not as columns.
- Bare `number` produces a `KYSELY_AMBIGUOUS_NUMBER` diagnostic and the column is skipped, requiring an explicit `int`/`bigint`/`float`/`decimal`/`money` type.
- Skips non-SQL store targets (`memory`, `localStorage`, `mongodb`, `dynamodb`, `eventSourced`).
- Supports `dialect` (`postgresql`, `mysql`, `sqlite`), `tableMappings`, `columnMappings`, `typeMappings`, and `emitFactory` options.

```ts
import { getProjection } from '@angriff36/manifest/projections';

const projection = getProjection('kysely');
const result = projection.generate(ir, {
  surface: 'kysely.schema',
  options: { dialect: 'postgresql' },
});
```

### Terraform

The Terraform projection generates HCL resource definitions from Manifest IR for database tables, storage buckets, and pub/sub topics. It supports three cloud providers: **AWS** (emits `aws_db_instance`, `aws_s3_bucket`, `aws_sns_topic`), **GCP** (emits `google_sql_database_instance`, `google_storage_bucket`, `google_pubsub_topic`), and **Supabase** (emits `supabase_project` with documented table schemas).

The projection is registered under the name `terraform` and emits one surface: `terraform.main` (content type `hcl`). It supports both single-file (`main.tf`) and multi-file (`database.tf`, `storage.tf`, `messaging.tf`) output modes.

**Key characteristics:**

- Walks `ir.stores` to find entities with persistent store targets (`postgres`, `supabase`) for database resources.
- Walks `ir.events` to emit pub/sub topic definitions.
- Accepts `storageBuckets` in options for S3/GCS bucket definitions.
- Skips non-persistent targets (`memory`, `localStorage`, `durable`) with diagnostic codes.
- Emits proper HCL with primary keys, unique constraints, `NOT NULL`, and encryption settings.

```ts
const projection = getProjection('terraform');
const result = projection.generate(ir, {
  surface: 'terraform.main',
  options: { provider: 'aws', outputMode: 'multi-file' },
});
```

### Materialized View SQL

The materialized views projection generates PostgreSQL `CREATE MATERIALIZED VIEW` DDL from Manifest IR entities. It supports three refresh strategies: **on-demand** (emits `REFRESH MATERIALIZED VIEW`), **scheduled** (emits `SELECT cron.schedule(...)` for pg_cron), and **trigger-based** (emits `CREATE TRIGGER ... AFTER UPDATE OF col ... EXECUTE FUNCTION`).

The projection is registered under the name `materialized-views` and emits one surface: `materialized-views.ddl` (content type `sql`).

**Key characteristics:**

- Includes an expression-to-SQL translator that converts `IRExpression` trees to PostgreSQL: literals, identifiers, binary operations (`===` maps to `=`), function calls (`SUM`, `COUNT`), and conditionals (`CASE WHEN`).
- Supports schema-qualified view names and `CREATE [UNIQUE] INDEX` statements (btree, hash, gin, gist).
- All relational concepts (view names, column aliases, refresh strategies, index definitions) enter via projection options, never into the IR.
- Deterministic output for a given IR + options pair.

```ts
const projection = getProjection('materialized-views');
const result = projection.generate(ir, {
  surface: 'materialized-views.ddl',
  options: {
    views: [{
      name: 'order_summary',
      source: 'Order',
      strategy: 'scheduled',
      schedule: { cron: '0 * * * *' },
      columns: [{ expression: 'SUM(total)', alias: 'revenue' }],
    }],
  },
});
```

### Analytics Event Schema

The analytics projection generates typed event schemas from Manifest IR for Segment, Amplitude, Mixpanel, or Snowplow. It produces three surfaces:

- `analytics.tracking-plan` — A JSON tracking plan document following the Segment Tracking Plan spec format, with events derived from command emits and standalone event declarations.
- `analytics.events` — A TypeScript module with typed property interfaces per event, `AnalyticsEvents` constants, `AnalyticsEventMap` type mapping, and provider-specific `track()` functions.
- `analytics.handlers` — TypeScript command handler stubs that inject typed `track()` calls after command execution.

The projection is registered under the name `analytics` and supports the `provider` option (`'segment'` | `'amplitude'` | `'mixpanel'` | `'snowplow'`, default `'segment'`). Events are deduplicated by name across command emits and standalone declarations. All output is deterministic with sorted iteration.

```ts
const projection = getProjection('analytics');
const result = projection.generate(ir, {
  surface: 'analytics.tracking-plan',
  options: { provider: 'segment' },
});
```

### Elasticsearch / OpenSearch

The Elasticsearch projection generates index mappings, templates, ingest pipelines, an outbox-driven indexer worker, and a typed search client from Manifest IR entities stored in Elasticsearch. It emits five surfaces:

- `elasticsearch.mapping` — Per-entity ES index mappings with proper field types, settings, and analyzers.
- `elasticsearch.indexTemplate` — Composable index templates with index patterns and priority.
- `elasticsearch.ingestPipeline` — Ingest pipeline processor configurations.
- `elasticsearch.indexer` — TypeScript worker that consumes outbox entries and bulk-indexes entity state changes with retry logic and exponential backoff.
- `elasticsearch.client` — TypeScript search query builders with `multi_match` full-text search and typed document interfaces.

The projection is registered under the name `elasticsearch`. It maps Manifest types to ES field types (string to `keyword`/`text`, int to `integer`, float to `scaled_float`, etc.). Bare `number` produces an `ELASTICSEARCH_AMBIGUOUS_NUMBER` diagnostic. Computed properties are excluded from index mappings by default. Multi-tenant support automatically adds `tenantId` as a required `keyword` field when tenant context is configured.

```ts
const projection = getProjection('elasticsearch');
const mapping = projection.generate(ir, { surface: 'elasticsearch.mapping' });
const indexer = projection.generate(ir, { surface: 'elasticsearch.indexer' });
```

### Storybook

The Storybook projection generates Component Story Format (CSF3) stories from Manifest IR entities and commands. It emits three surfaces:

- `storybook.entity` — Default stories for each entity with all properties mapped to Storybook controls (text, number, boolean, date, select for enums). Computed properties are rendered as `control: false`. Private properties are excluded.
- `storybook.command` — Interaction stories with `GuardsPass` and `GuardFails` scenarios using play functions. Guard expressions are analyzed heuristically to produce meaningful pass/fail args.
- `storybook.all` — Combined entity and command stories in a single artifact.

The projection is registered under the name `storybook`. It supports `componentImportPattern` (with `{Entity}` and `{Command}` placeholders), `titlePrefix`, `includeGuardScenarios`, and `includeConstraintStories` options. Constraint violation stories are generated for entities with constraints.

```ts
const projection = getProjection('storybook');
const stories = projection.generate(ir, {
  surface: 'storybook.all',
  options: { titlePrefix: 'Manifest/' },
});
```

### Python Client SDK

The Python Client SDK is delivered through the `pydantic.client` surface of the [Pydantic projection](/projections/python-pydantic). It generates a `ManifestClient` class with `httpx.AsyncClient` integration, entity query methods (`list_<entity>s`, `get_<entity>`), typed command invocation methods, optional bearer token auth, and module-level convenience functions. The generated code targets Python 3.10+.

```ts
const projection = getProjection('pydantic');
const client = projection.generate(ir, {
  surface: 'pydantic.client',
  options: { clientBaseUrl: 'https://api.example.com', emitConvenienceFunctions: true },
});
```

## 📄 Manifest doc — `language/modules` · Modules: multi-file projects and imports

Manifest supports multi-file projects through `use` declarations and `import` statements. The module resolver discovers all referenced files, detects cycles, sorts them topologically, and merges the compiled IR into a single deterministic output. This lets you split large programs into focused files without sacrificing whole-program validation.

## Use declarations

The `use` keyword includes another `.manifest` file. Use declarations must appear at the top of the file, before any entity, event, or store declarations:

```manifest
// app.manifest
use "./shared/types.manifest"
use "./domains/order.manifest"
use "./domains/product.manifest"
```

The resolver walks `use` declarations recursively to discover the full file graph. A `use` path must be relative and end in `.manifest` — absolute paths and extensions other than `.manifest` produce a parse error.

## Import statements

For selective imports, use the `import` syntax to bring specific symbols into scope:

```manifest
import { Status, Priority } from "./shared/types.manifest"

entity Article {
  property required title: string
  property status: Status = draft
  property priority: Priority = medium
}
```

Import statements support aliasing:

```manifest
import { User as Customer } from "./shared/user.manifest"

entity Order {
  property required buyer: Customer
}
```

The resolver validates import specifiers against the source file's exported symbols (entities, enums, and value objects). Importing a symbol that does not exist, or importing a symbol of the wrong kind, produces a compile error diagnostic.

## Compilation pipeline

<Steps>
  <Step title="Resolution">
    Walk all `use` and `import` declarations starting from the entry file. Build a complete dependency graph using BFS discovery.
  </Step>
  <Step title="Cycle detection">
    Detect circular imports (A imports B imports A) using DFS with grey/black coloring. Circular dependencies produce a compile error diagnostic listing the cycle path.
  </Step>
  <Step title="Topological sort">
    Kahn's algorithm determines the compilation order — dependencies compile before dependents. Ties are broken alphabetically for deterministic output.
  </Step>
  <Step title="Per-file compilation">
    Each file is parsed and compiled to IR independently, producing one IR per file.
  </Step>
  <Step title="Cross-file validation">
    The merged IR is validated for cross-file reference integrity: entity types referenced in properties must exist, enum members used as defaults must be declared, and event names referenced in `emit` must be defined somewhere in the project.
  </Step>
  <Step title="IR merging">
    All per-file IRs are merged into a single output with sorted arrays for determinism. A `sources` array in the IR provenance tracks which files contributed to the merged output.
  </Step>
</Steps>

## CLI flags

Use the `--merge` and `--entry` flags to compile multi-file projects:

```bash
# Compile a multi-file project with merging
manifest compile --merge --entry src/app.manifest

# Compile with explicit output path
manifest compile --merge --entry src/app.manifest --output ir/

# Let the compiler auto-detect entry files (files not referenced by any other)
manifest compile --merge src/
```

| Flag | Description |
|------|-------------|
| `--merge` | Enable multi-file compilation and merge all discovered files into a single IR output |
| `--entry <path>` | Specify the entry point for module resolution. When omitted, the compiler auto-detects root files. |

## Namespace isolation

Each module's entities, events, and enums are namespaced by the module's file path. Two files can define entities with the same name without collision:

```
shared/types.manifest  ->  Status enum  ->  shared/types.Status
domains/order.manifest ->  Order entity ->  domains/order.Order
domains/product.manifest -> Order entity -> domains/product.Order
```

When using `import { X }` syntax, the imported symbol is available without the module prefix in the importing file. The `use` syntax makes all declarations available but with the module prefix for disambiguation.

<Note>
Cross-file validation catches ambiguous references when two modules export the same symbol name. Use the module prefix or an import alias to disambiguate.
</Note>

## Cross-file validation

After merging, the compiler validates cross-file references and produces error diagnostics with file paths and line numbers:

- Entity types referenced in properties must exist in the merged IR
- Enum members used as default values must be declared in an imported or used file
- Event names referenced in `emit` actions must be declared somewhere in the project
- Store declarations must reference entities that exist in the merged IR

Invalid cross-file references produce compile error diagnostics rather than runtime failures.

## ResolverHost abstraction

The module resolver uses a `ResolverHost` abstraction for file system access. In production, this reads from disk. In tests, you can provide an in-memory host to test resolution without touching the file system:

```typescript
import { ModuleResolver } from "@angriff36/manifest/module-resolver";

const resolver = new ModuleResolver({
  readFile(path) {
    // Return file contents from an in-memory map
    return mockFiles.get(path);
  }
});
```

This makes the resolver testable in isolation and allows the multi-compiler to work in environments without direct file system access.

## Complete example

A multi-file project with shared types, domain entities, and an entry point:

```manifest
// shared/types.manifest
enum Status { draft, submitted, approved, rejected }
enum Priority { low, medium, high }
```

```manifest
// domains/order.manifest
use "../shared/types.manifest"

entity Order {
  property required id: string
  property status: Status = draft
  property priority: Priority = medium
  property amount: number = 0

  command submit() {
    guard self.status == draft
    mutate status = submitted
    emit OrderSubmitted
  }
}

event OrderSubmitted: "order.submitted" {
  orderId: string
}

store Order in memory
```

```manifest
// app.manifest
use "./domains/order.manifest"

// Additional top-level declarations can go here
```

Compile the project:

```bash
manifest compile --merge --entry src/app.manifest --output project.ir.json
```

The resulting `project.ir.json` contains all entities, events, enums, and stores from every discovered file, merged into a single deterministic IR with provenance tracking.

See [Entities](/language/entities) for entity declarations, [Events](/language/events) for event schemas, and [CLI overview](/cli/overview) for the full set of compile options.
