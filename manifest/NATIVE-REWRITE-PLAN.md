## WS0 ‚Äî Empty-string UUID defaults (‚õî CRITICAL, do FIRST ‚Äî active create-breaking bug)

**What:** 188 `uuid ‚Ä¶ = ""` defaults across 69 source files. An empty string is not a UUID; when a create persists such a field it hits a `@db.Uuid` column and Postgres rejects it (`invalid input syntax for type uuid: ""`). Hidden until now by the 3.3.0 composite-id regression (creates never reached the DB); the 3.3.1 fix exposed all 188.

**Why FIRST:** every governed create touching one of these fields is currently broken (verified: `TrainingAssignment.create` ‚Üí `employeeId=""`; `Notification.create` succeeds only because it has no such field). This blocks real user flows today ‚Äî it is not modernization, it is a live regression surface.

**The persist-before-mutate wrinkle (important ‚Äî a blind `s/= ""//` is NOT sufficient):** the engine inserts the row from the create *bootstrap* (defaults + same-named params) and runs the command's `mutate` actions *afterward* as an update. So a field whose value is supplied by a `mutate` from a **differently-named param or context** (`mutate employeeId = staffMemberId`, `mutate assignedBy = user.id`) is NOT populated at the first insert ‚Äî it lands on its default.

> **PROVEN (2026-07-07, engine store-op trace of `TrainingAssignment.create`):** `[ ["CREATE", ""], ["UPDATE", "c4a2402f-‚Ä¶"] ]`. The INSERT carries `employeeId=""` (default; param is `staffMemberId`, field is `employeeId` ‚Äî names differ, so no bootstrap seed), then a second UPDATE writes the real value. In-memory stores accept the `""` insert then update (unit tests pass, hiding the bug); Postgres `@db.Uuid` rejects the `""` INSERT before the UPDATE runs. Contrast `TimeOffRequest.create`, whose param IS `employeeId` (matches the field) ‚Üí filled at insert, no separate update needed. **Rule of thumb: a create param that feeds a `@db.Uuid`/required field via a rename-mutate (`field = otherParam`) or context (`= user.id`) is broken against Postgres.**

Two cases:
- **Nullable UUID (`uuid? = ""`):** drop the `= ""` ‚Üí inserts as `null` (valid), the later `mutate` fills it. **IR-only change ‚Äî NO migration** (verified 2026-07-11, compiler 3.4.25 + post-`20260710142245_reconcile_schema_truth`). The prisma projection no longer emits `@default("")` on `@db.Uuid` columns ‚Äî bare `rg '@default("".*@db.Uuid|@db.Uuid.*@default("")' packages/database/prisma/schema/manifest.prisma` ‚Üí 0 matches (the 614 `@default("")` lines are all String/Text/array). So dropping the source `= ""` removes only the IR `defaultValue` block; `manifest.prisma` is byte-identical and `db:check` stays clean. **129 of 186 fields are nullable ‚Üí these are fast Phase-1-style batches** (edit ‚Üí `manifest:build` ‚Üí `manifest:ci` ‚Üí commit; no `db:dev`, no "awaiting deploy"). Proven end-to-end on `KnowledgeBaseEntry.authorId` (`manifest:ci` green, zero-line `manifest.prisma` diff). The earlier "needs a migration (drop default)" below was true pre-3.4.22; it is stale now.
- **Required/NOT-NULL UUID set only by a mutate (e.g. `assignedBy`):** null/`""` both fail at insert. Fix at the source so the value is present at bootstrap ‚Äî rename the create param to match the field (so it seeds), OR accept the field as a create param, OR make the column nullable. Decide per field; don't auto-null a NOT-NULL column. **57 of 186 fields are required (`uuid = ""`)** ‚Äî these still need case-by-case persist-before-mutate analysis and MAY touch schema (nullability) ‚Üí migration.

**‚úÖ PILOT DONE (2026-07-07, `TrainingAssignment`):** `employeeId: uuid? = "" ‚Üí required uuid` with the create param renamed `staffMemberId ‚Üí employeeId` (same-named ‚Üí seeds the INSERT); `assignedBy: uuid ‚Üí uuid?` (filled by the post-insert `mutate assignedBy = user.id` UPDATE ‚Äî proven to persist). Migration `20260707125708_training_assignment_uuid_not_null` (repair-diff trimmed to the two ALTER clauses; `migrate dev` unusable while the accepted drift residual stands ‚Äî use `pnpm db:repair` + trim + `db:deploy` per batch).
