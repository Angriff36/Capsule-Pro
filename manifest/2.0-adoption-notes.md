## A. Authoring syntax — extracted from the installed 2.0.6 PARSER (authoritative)
Source: `manifest/runtime/node_modules/@angriff36/manifest/dist/manifest/parser.js`
(`parseReaction`, `parseApproval`, `parseApprovalStage`, `parseSaga`, `parseSagaStep`). The public
mintlify docs do NOT document these — the parser is the source of truth for source syntax.

### Reaction (top-level block)
```
on <EventName> run <Entity>.<command>
  resolve <expr>
  params { <name>: <expr>, <name>: <expr> }   # optional
```
- `on` + event identifier; `run` + `Entity.command`; `resolve <expr>` (required — resolves the target
  instance id); optional `params { name: expr, ... }`. IR node `{ type:'Reaction', event, targetEntity,
  targetCommand, resolve, params? }`.

### Approval (top-level block)
```
approval <Name> {
  command: <commandName>
  stages {
    <stageName> { policy: <expr>  when: <expr>?  required: <number>? }
    ...
  }
  timeout: <number>?            # numeric
  on_timeout: cancel | escalate # bare/string
  emit <EventName>              # optional, repeatable
}
```
- `command:` is REQUIRED (throws otherwise). `stages { ... }` holds `stage <name> { }` or bare
  `<name> { }`. Each stage REQUIRES `policy:` (expr); optional `when:` (expr gate) and `required:`
  (count, default 1). IR node `{ type:'Approval', name, command, stages[], emits[], timeout?, onTimeout? }`.

### Saga (top-level block)
```
saga <Name> {
  step <stepName> { command: <Entity>.<command>  compensate: <Entity>.<command>? }
  ...
  on_failure: compensate | abort   # default 'compensate'
  emit <EventName>                 # optional, repeatable
}
```
- `step` is an IDENTIFIER (not keyword). Each step REQUIRES `command: Entity.command`; optional
  `compensate: Entity.command`. Must have ≥1 step. IR node `{ type:'Saga', name, steps[], onFailure, emits[] }`.

## E. Phase 0 — RUNTIME EXECUTION FINDINGS (read of runtime-engine.js, 2026-06-02)
Read `manifest/runtime/node_modules/@angriff36/manifest/dist/manifest/runtime-engine.js`.
These are the load-bearing facts that decide what can actually be wired vs what would dangle.

### Emits live in `command.emits` (NOT actions)
IR `command.emits: string[]` (950/952 commands have it). `Event.create.emits = ["EventCreated"]`
(`event-rules.manifest:85`). `EventCreated` payload = eventId, tenantId, title, eventType, eventDate,
guestCount, clientId, createdAt. Earlier probe that read `actions` for emits was wrong.

### ⚠ REACTIONS auto-dispatch INSIDE runCommand, but CREATE-reactions silently no-op
- Engine (L2015-2065): after a command's events emit, it iterates `ir.reactions`, matches by
  `r.event === emitted.name`, evaluates `resolve`/`params` against `{payload, self}` (= event payload),
  and dispatches `runCommand(targetCommand, input, { entityName, instanceId: String(resolvedId) })`
  with a depth guard (MAX_REACTION_DEPTH=10). So reactions need NO host wiring — authoring them is enough.
- **BUT the reaction ALWAYS sets `instanceId`**, and autoCreate only fires when `!instanceId`
  (§15d: `shouldAutoCreateInstance = name==='create' && entityName && !instanceId`). ⟹ **a reaction
  whose target is a `create` command is a NO-OP** — mutates run against a non-existent instance →
  nothing persists. The marketed "EventCreated → create Proposal/Budget/Tasks" pattern DOES NOT WORK
  in 2.0.6. Reactions are viable ONLY for transitions/mutations on an EXISTING instance whose id is
  derivable from the event payload (e.g. `on PaymentReceived run Invoice.markPaid resolve self.invoiceId`).

### ✅ SAGAS are fully viable (create AND transition steps)
- `runSaga(name, stepInputs, options)` (L1360) runs steps in order; on failure with
  `on_failure: compensate`, compensates completed steps in REVERSE (best-effort, L1429). Self-contained
  in ONE runtime call → no cross-request state needed.
- Per-step dispatch (L1378): `instanceId: cfg.instanceId` comes from `stepInputs[step.name]`. So the
  CALLER controls instanceId per step → omit it for a `create` step (autoCreate fires) or pass it for a
  transition step. **Sagas avoid the reaction create-trap.** Caveat (d.ts): in-memory saga is NOT
  transactional — partial failure relies on compensation, not DB rollback.

### ⚠ APPROVALS are in-memory only + a policy hack — NOT production-wireable as-is
- `checkApprovalGate` (L3108) is called from runCommand (L1875); returns `CommandResult` with
  `approvalRequired` when stages are pending, else proceeds. Stages with a `when` expr only count when
  true; `required:` is a grant count.
- **State is `this.approvalRequests = new Map()` (L210), in-memory on the engine instance. There is NO
  `approvalStore` option** (RuntimeOptions has idempotency/audit/outbox/jobQueue stores only). Capsule
  builds a fresh engine per HTTP request → manager-now/finance-later **will not persist**. No bulk
  hydrate/import method is exposed (only `requestApproval`/`approveStage`/`denyApproval`/`getApprovalRequest`).
- `approveStage` (L3232) evaluates the stage `policy` with `user: { id: approverUserId, role:
  approverUserId }` — comment: "userId doubles as role for testing." So real RBAC policies eval against
  a hack context. ⟹ Approvals need a Capsule persistence shim + role-context fix before they are real.

## F. Phase evidence log

### SAGA pilot — `ProcessInvoicePayment` (Payment.process → Invoice.applyPayment)
**Authored:** `manifest/source/payment-rules.manifest` (top-level `saga` block).
**Merge-gap fix (KEYSTONE):** `manifest/scripts/ir-utils.mjs` `mergeIrs` did NOT carry top-level
`sagas`/`reactions` — per-file IR had them, the merged IR dropped them. Every saga/reaction authored
in source was silently vanishing. Added `sagasResult`/`reactionsResult` (same dedup pattern) to the
merge output + totals. (Approvals ride on `entity.approvals` → already merge with their entity.)
**Compiled:** `pnpm manifest:compile` → merged `kitchen.ir.json` now has `sagas:1` (verified).

**Executed roundtrip** (`.tmp/saga-roundtrip.mjs`, real merged IR + real `RuntimeEngine`, in-memory
Store, 2026-06-02). NOTE 2.0.6 change: durable entities REQUIRE a `storeProvider` (no default memory
fallback) — pass it as the 3rd ctor arg; `MemoryStore` is NOT exported (inline a Store). Policy gate:
`EventManagement` requires `user.role in [event_coordinator,manager,admin]` AND `user.tenantId != ""`
(set tenantId ON the user object).
```
SCENARIO 1 (happy path):
  runSaga -> {success:true, status:"completed", steps:[processPayment:completed, applyToInvoice:completed]}
  payment.status PENDING->COMPLETED | invoice SENT->PARTIALLY_PAID, amountPaid 0->100, amountDue 100->0
SCENARIO 2 (step2 paymentAmount>amountDue, forced fail):
  runSaga -> {success:false, status:"compensated", failedStep:"applyToInvoice",
              steps:[processPayment:compensated, applyToInvoice:failed(Guard condition failed)]}
  after: payment.status STILL COMPLETED (NOT refunded), invoice rolled back to SENT/0/100
```
⟹ **Happy-path saga orchestration WORKS and persists.** ✅
⟹ **Compensation is COSMETIC for parameterized compensations** (CONFIRMED): engine calls
`runCommand(compensate, {}, ...)` (empty input, engine L1444); `Payment.refund` needs `refundAmount`
→ guard fails → swallowed best-effort → step mislabeled "compensated" though no refund happened.

### DOD status for the saga
1 authored ✅ · 2 in merged IR ✅ · 3 runtime-honors ✅ (proven) · 5 proven ✅ (above) ·
4 reachable from a route ❌ NOT YET · 6 reconcile ❌.
