## 1. What This Is

Plain-English purpose:

```text
Generates TypeScript interfaces (one per entity), string-literal-union enums (one per status field), and typed command-input interfaces from the compiled Manifest IR. The output is a single barrel file consumed by app code for type-safe entity shapes and command payloads.
```

Real app impact:

```text
When correct:
- App code, generated clients, forms, commands, and tests share one canonical entity shape.
- Agents stop inventing duplicate local interfaces.
- Stale generated code becomes visible in CI (manifest:react-query:check catches hook drift;
  manifest:contract:check blocks direct imports into generated artifacts).

When wrong:
- Types drift from Manifest IR.
- Agents add sloppy null guards and fake isRecord-style checks.
- Forms and command payloads compile while being wrong at runtime.
- Generated files may exist but not be the actual source used by the app.
```

## 2. Ryan Final Decision

Decision:

```text
NEEDS-RYAN
```

Reason:

```text
Type generation is working and wired. Open questions remain around:
(1) whether types should be in a shared package vs app-local,
(2) whether there should be a dedicated type-staleness CI gate (today only hooks and openapi have explicit gates),
(3) whether the audit-contract-imports allowlist correctly covers type imports.
```

Do not do:

```text
Do not hand-write duplicate Manifest entity, command, or event types in random app folders.
Do not create local fallback types because generated types are inconvenient to import.
Do not wire app code to stale generated output.
Do not import directly from manifest/generated/ or manifest/ir/ — blocked by manifest:contract:check.
```

## 3. Current Status

Current recorded status:

```text
Type generation is ACTIVE and produces a single 3898-line barrel file containing 200+ entity interfaces
and 17 status enums. Generated by manifest:client (generate-capsule-client.mjs). Consumed via
`import { EventType } from "@/app/lib/manifest-types.generated"`. No dedicated type-staleness CI
gate exists; staleness is caught indirectly via react-query:check and contract:check.
```

Known gaps:

```text
1. No dedicated type-staleness CI gate — if types drift from IR without hooks or openapi also drifting,
   CI will not catch it.
2. manifest-types.generated.ts and manifest-client.generated.ts are DIFFERENT files but share
   the same generation header comment line 1 ("GENERATED by manifest:client").
3. @repo/manifest-runtime (constitution §4a) lives at `manifest/runtime/` — the `packages/manifest-runtime/`
   path was retired by §19a (verified: `manifest/runtime/package.json` name === `@repo/manifest-runtime`).
   The import boundary is additionally enforced by `audit-contract-imports.mjs` (`pnpm manifest:contract:check`).
4. Zod schemas are generated to manifest/generated/schemas/ but it is UNKNOWN whether app code imports
   them directly (blocked by contract:check if so).
```

## 8. Canonical Behavior

Happy path:

```text
1. Manifest source files (*.manifest) are compiled to IR (manifest/ir/kitchen.ir.json).
2. pnpm manifest:client reads IR, maps scalars to TS types, and emits
   apps/app/app/lib/manifest-types.generated.ts.
3. App code imports types from the generated barrel.
4. manifest:ci gates catch indirect staleness via react-query:check and contract:check.
```

Failure behavior:

```text
If generation fails, types may be stale. No dedicated type-staleness gate exists — drift is caught
only if it cascades to hooks or openapi output. If types drift without affecting those projections,
CI will pass with stale types.
```

Forbidden behavior:

```text
No duplicate hand-written app types for Manifest entities.
No random local interfaces for Manifest command payloads.
No imports from manifest/generated/, manifest/ir/, or manifest/api-docs/ in app code
(blocked by manifest:contract:check).
```
