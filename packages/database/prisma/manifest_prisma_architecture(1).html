Manifest + Prisma map

Manifest is the app behavior boundary. Prisma is the database plumbing underneath it. The map below shows the actual paths: allowed runtime path, generated projection path, internal Prisma path, and the direct Prisma bypass path that causes drift.

Legend

Allowed runtime path: app calls Manifest; Manifest decides; storage persists.
Generated/projection path: Manifest source generates app/client/schema outputs.
Internal Prisma path: PrismaClient is allowed only in storage adapter plumbing.
Bypass path: app code imports PrismaClient directly and skips Manifest.

The actual architecture map

Green is the runtime path users should trigger. Blue is what gets generated from Manifest. Amber is where PrismaClient belongs. Red is what should be blocked or deleted from app code.

What the map means in plain English

App code calls Manifest.

Pages/forms should trigger generated Manifest commands. That keeps behavior centralized.

PrismaClient is not deleted.

It stays inside the Prisma adapter/store so approved Manifest changes can persist to Postgres.

Direct Prisma from app code is the problem.

It skips command rules, policy checks, tenant enforcement, validation, and emitted events.

StaffPerformance example

Layer What it does for StaffPerformance What breaks if bypassed
Manifest Defines commands like createReview, completeReview, acknowledgeReview; owns allowed status flow. Reviews can jump to invalid states or skip required acknowledgement/completion behavior.
Generated app/API Gives UI and routes a standard way to call the Manifest command. Agents invent one-off CRUD paths and the app stops behaving consistently.
Prisma schema Maps app names to DB names, such as tenantId to tenant_id and model to staff_performances. Wrong table/column names, duplicate models, migration drift.
PrismaClient Used internally by the Manifest store to write/read the approved row. If used directly by app code, Manifest never sees the change.
Postgres Stores the durable row. Database becomes the only truth, while app behavior becomes scattered.

Allowed

PrismaClient import in storage plumbing only.

Not allowed

PrismaClient import in app behavior files.

apps/app/**/page.tsx apps/app/**/actions.ts apps/app/**/components/** feature-specific route.ts files random CRUD helper files

The one-sentence rule

Manifest decides. Prisma persists.

Keep PrismaClient behind Manifest, and Prisma becomes useful database plumbing instead of a competing source of app behavior.
